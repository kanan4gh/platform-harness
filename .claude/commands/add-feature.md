# 新機能の追加 (計画承認後・完全自動実行モード)

> **手順の正は `docs/procedures/add-feature.md`(ハーネス中立の手順書)。本コマンドはそのClaude Code最適化版であり、差分はツール割当(worktree/Explore/Plan/プランモード/ビルトインスキル/サブエージェント)のみ。手順の骨格を変更する場合は手順書→本コマンドの順で更新すること。**

**重要:** このワークフローの承認ゲートは**ステップ4.5(計画の承認)の1箇所のみ**です。そこまでは調査・計画を進めて承認を求め、**承認された後は、ユーザーの介入なしに完了まで完全に自動で実行**されます。承認後の各ステップは完了後ただちに次のステップへ移行し、途中でユーザーに確認を求めたり作業を中断したりしないでください(「計画は承認必須・実装は自動」の原則)。

**引数:** 機能名 (例: `/add-feature ユーザープロフィール編集`)

---

## ステップ0: 作業の隔離 (worktree + フィーチャーブランチ)

mainの作業ツリーを汚さないための機構的な担保。**このステップ以降の全作業はworktree内で行う。**

1. 関連するGitHub Issueを確認する。存在しない場合は `gh issue create` で作成し、Issue番号を控える。
2. 機能名から**英語 kebab-case のタスク名**を決める(例: 「ユーザープロフィール編集」→ `user-profile-edit`)。ステアリングディレクトリ名(`.steering/[日付]-[タスク名]/`)にも同じタスク名を使う(repository-structure.md の命名規則)。
3. worktreeとフィーチャーブランチを作成する:
  - `EnterWorktree` ツールが利用可能な場合はそれを使用する(ブランチ名: `feature/[タスク名]`)。
  - 利用できない場合は以下を実行し、以降の作業ディレクトリを移す:
    ```bash
    git worktree add ../[リポジトリ名]-[タスク名] -b feature/[タスク名]
    ```
4. worktree作成に失敗した場合(ブランチ重複等)はユーザーに報告して中断する。自動リネーム等で暗黙に進めない。

**フォールバック**: worktreeが使えない環境では `git checkout -b feature/[タスク名]` の通常ブランチで代替してよい。ただし**mainブランチ上で実装を始めないこと**だけは必ず守る。

## ステップ1: 準備とコンテキスト設定

1. 現在のタスクコンテキストを確立する:
  - 機能名: `[引数で与えられた機能名]`
  - 日付: `[現在の日付をYYYYMMDD形式で取得]`
  - ステアリングディレクトリパス: `.steering/[日付]-[機能名]/`
2. 上記ステアリングディレクトリを作成する。
3. ステアリングファイルはステップ4のパス判定後に作成する(通常パス: requirements.md / design.md / tasklist.md の3ファイル、軽量パス: design.md を省略した2ファイル)。

## ステップ2: プロジェクト理解

1. `AGENTS.md`(SDDプロセス正典)と `CLAUDE.md`(Claudeアダプタ)を読み、プロジェクトの全体像を把握する。
2. `docs/`ディレクトリ内の永続ドキュメントを確認し、関連する設計思想やアーキテクチャを理解する。

## ステップ3: 既存パターンの調査 (Exploreエージェント委譲)

1. `Task`ツールでビルトインの `Explore` エージェントを起動し、調査を委譲する。
  - `subagent_type`: "Explore"
  - `model`: "sonnet" (モデル配分方針: 調査はトークン消費が大きく判断は浅いため、メインより軽いモデルに委譲する。`.claude/README.md` のモデル配分表を参照)
  - `prompt`: "「[機能名]」に関連する既存の実装パターン・命名規則・類似コンポーネントを調査してください。検索幅はmediumで。結論(発見したパターンの一覧と参照ファイルパス)のみを返してください。"
2. Exploreの結論を、設計(ステップ4)の入力として使う。

**フォールバック**: `Explore` エージェントが使えない環境では、従来どおりメインエージェントがGrepツールで検索・分析する。

## ステップ4: 計画フェーズ (Plan活用 + ステアリングファイルの自動生成)

1. `Task`ツールでビルトインの `Plan` エージェントを起動し、実装戦略の立案を委譲する。
  - `subagent_type`: "Plan"
  - `prompt`: "「[機能名]」の実装計画を立案してください。ステップ3の調査結果: [Exploreの結論]。ステップごとの作業・変更対象ファイル・アーキテクチャ上のトレードオフを返してください。"
2. **パス判定**: 手順書(`docs/procedures/add-feature.md` ステップ4)の適用基準4項目をすべて満たす場合、軽量パスを選択できる(軽量パスではステップ2-3の読み込み・調査を関連ドキュメント・関連コードに限定してよい)。
3. **G3受け入れの要否判定**: 手順書(`docs/procedures/add-feature.md` ステップ4)の基準で判定する。基準は「**ハーネスのアダプタ構成・権限・フックを変更したか**」。Claude Codeでは `.claude/skills/` `.claude/agents/` `.claude/commands/` の**frontmatter・ファイル名・配置**、`.claude/settings.json` の権限、`.claude/hooks/` の定義・登録を変えた場合にG3が必要。**コマンド・スキル本文の手順記述だけを書き換えた変更や、手順書・テンプレート・`docs/`・テストのみの変更では不要**。判定結果を `requirements.md` に記録する。
4. `Skill('steering')`を**計画モード**で実行し、Planの出力を `design.md` の下敷きとして、ステアリングファイルの内容を生成する(通常パス: 3ファイル、軽量パス: requirements.md に「**軽量パス**: 適用」と判定結果を記録した2ファイル)。
  - `tasklist.md` のフェーズは、テンプレート(`docs/procedures/templates/tasklist.md`)の「フェーズとステップの対応」に従って**消化するステップごとに排他的に分割する**(実装フェーズ=ステップ5 / 4段検証フェーズ=ステップ6 / 振り返りとドキュメント更新フェーズ=ステップ7 / 最終品質ゲートフェーズ=ステップ8)。
5. このステップが完了したら、ステップ4.5(計画の承認)に進む。

**フォールバック**: `Plan` エージェントが使えない環境では、メインエージェントが直接設計する。

## ステップ4.5: 計画の承認 (このワークフロー唯一の承認ゲート)

生成した計画をユーザーに提示し、機構による承認を得る。

1. `EnterPlanMode` でプランモードに入る。
2. `design.md`(軽量パスでは `requirements.md`)の要点を計画として提示する: 実装アプローチ / 変更対象ファイル / tasklist.mdのタスク一覧 / 検証方針 / **G3受け入れの要否判定**。**軽量パスを選択した場合は、適用判定(基準4項目の判定結果)を計画と一緒に提示する**(暗黙に軽量化しない)。
3. `ExitPlanMode` で承認を求める。
4. **承認された場合**: 以降は無停止で実行する。ただちにステップ5へ進む。
5. **修正を指示された場合**: 指示をステアリングファイルに反映し、再度この承認ゲートを実行する。

**フォールバック**: プランモードが使えない環境では `AskUserQuestion`(選択肢: 「承認して実装開始」「修正を指示する」「中止する」)で代替する。それも使えない場合はテキストで承認を求めて待機する。

## ステップ5: 実装ループ (実装フェーズの完全消化)

**このステップは、`tasklist.md`の実装フェーズの全タスクが `[x]` になるまで自動で繰り返されるループ処理です。**
**このステップが正常に完了したら、決して停止せず、ただちにステップ6に進むこと。**

**このループが消化するのは実装フェーズだけです。** 4段検証フェーズ・振り返りとドキュメント更新フェーズ・最終品質ゲートフェーズのチェックボックスは、それぞれステップ6 / 7 / 8が消化します。**これらをステップ5で先行して消化してはいけません**(先行消化すると、正式なステップより前に検証・振り返り・ゲートを実行したことになり、最終状態の検証が成立しません)。最終品質ゲート行はステップ8まで `[ ]` で残る契約のため、終了条件を「全タスクが `[x]`」にするとループが終了できなくなります。

**ループ開始:**

1. タスクリストの読み込み:
  - `[ステアリングディレクトリパス]/tasklist.md`ファイルを読み込む。

2. 進捗の確認:
  - **実装フェーズに**未完了タスク (`[ ]`) が存在するか確認する。
  - **もし実装フェーズに未完了タスクが存在しない場合:** この実装ループは完了とみなし、ただちに**ステップ6**へ進む(後続フェーズの `[ ]` は残っていてよい)。
  - **もし存在する場合:** 次の処理（3. タスクの実行）に進む。

3. タスクの実行:
  - `tasklist.md`の**実装フェーズの先頭にある未完了タスク**を1つ特定する。
  - そのタスクを完了させるために必要な実装作業を実行する。
  - `Skill('steering')`を**実装モード**で使用する。
  - `Skill('development-guidelines')`のコーディング規約を常に遵守する。

4. タスクリストの更新:
  - 実行したタスクが完了したら、`Edit`ツールを使用して`tasklist.md`を更新し、該当タスクを `[ ]` から `[x]` に変更する。

5. ループ継続:
  - **ステップ5の先頭 (1. タスクリストの読み込み) に戻り、処理を繰り返す。**

---
### ※ 実装ループ内の例外処理ルール ※

実装ループの実行中に以下の状況が発生した場合は、このルールに従って自律的に対処し、ループを継続すること。

- **ルールA: タスクが大きすぎる場合**
  - **対処法:** 現在のタスクをより小さな複数のサブタスクに分割する。`Edit`ツールを使い、元のタスクを削除し、その場所に新しいサブタスク（`[ ]`付き）を挿入する。その後、ループを継続する。

- **ルールB: 技術的理由でタスクが不要になった場合**
  - **条件:** 実装方針の変更、アーキテクチャの変更、依存関係の変更など、明確な技術的理由がある場合にのみ適用。
  - **対処法:** `Edit`ツールを使い、該当タスクを `[x] ~~タスク名~~ (理由: [具体的な技術的理由を簡潔に記述])` の形式で更新する。その後、ループを継続する。

- **エスカレーション: 軽量パスの基準逸脱**
  - **条件:** 軽量パスで実装中に適用基準4項目のいずれかを満たさなくなった場合(4ファイル目の変更が必要になった / 新規アーキテクチャ要素・依存が必要と判明した / docs更新が必要と判明した / 破壊的変更が必要と判明した)。昇格条件はこの4項目の逸脱のみ。
  - **対処法:** 通常パスへ昇格する: design.md を追加作成し、requirements.md の軽量パス欄を「非適用へ昇格(理由)」に更新し、ステップ6は4段フルで実施する。**基準を破ったまま軽量パスで押し切ることを禁止する。**

- **❌ 絶対禁止の行為:**
  - 未完了タスクを「後でやる」「別タスクにする」などの理由で意図的にスキップすること。
  - 理由なく未完了タスクを放置してループを終了させること。
  - ユーザーに判断を仰ぐこと。

## ステップ6: 4段検証

`tasklist.md`の**実装フェーズの全タスクが `[x]` になったこと**を確認してから、以下を**段1→段2→(段3・段4は並列可)の順**で実行する。機械的に落ちるものを先に潰し、コストの高いLLM検証を後段に置く構成である。段3・段4は互いに独立しているため並列実行してよい。

**このステップが消化するのは4段検証フェーズのチェックボックスである。** 各段を実行し、その段の指摘対応まで終えた時点で該当行を `[x]` にする(段4の行は段4の指摘対応が完了した時点で `[x]` にする。段4の実行中に自分自身の行が `[ ]` であるのは正しい状態である)。振り返り・ドキュメント更新・最終品質ゲートの各フェーズはここでは消化しない。

**再検証は増分方式**: 各段の指摘を修正したら、再検証は変更の影響範囲に限定する(該当テスト・該当指摘の解消確認のみ。全段のやり直しはしない)。フル検証はステップ8の最終品質ゲートが決定論的に強制するため、中間のフル再実行は冗長。

**最終品質ゲート(`uv run python3 scripts/local_quality_gate.py`)はこのステップでは実行しない**: ステップ7で振り返りとドキュメント更新を行うため、ここで実行しても最終ファイル状態の検証にならない。ゲートはすべての変更を終えた後の終端ステップ(ステップ8)に置く。

**段3・段4は互いに独立しており、並列実行してよい。**

**軽量パスの縮約**: 段1・段2は必須。段3は同一セッションでの変更差分セルフレビューに縮約してよい(`Skill('code-review')` の起動は不要)。段4はスキップできる(スキップした旨と理由を `tasklist.md` に記録する)。

### 段1: 静的検証

1. 以下のコマンドを順番に実行し、すべてパスすることを確認する。
  ```bash
  Bash('uv run pytest')
  Bash('uv run ruff check')
  Bash('uv run basedpyright')
  ```
2. エラーが発生した場合は、問題を分析し、修正してから再度この段を実行する。

### 段2: 実挙動検証

1. 変更種別に応じて観察方法を選ぶ:

| 変更種別 | 観察方法 |
|---------|---------|
| 実行可能コード(フック・スクリプト等) | ビルトインの `Skill('verify')` で実際に起動し入出力を観察する |
| スキル・コマンド定義(Markdown) | 定義を実データに対して1回実行し、成果物を観察する |
| 純ドキュメント(docs/等のみ) | 観察対象がないため段2をスキップし、その旨を `tasklist.md` に記録する |

2. テスト通過だけで済ませず、必ず実挙動(または成果物)を観察すること。

### 段3: コードレビュー

1. ビルトインの `Skill('code-review')` を実行する。
2. 報告されたfindingsのうち正当なものを修正する(再検証は増分方式に従い、修正の影響範囲に限定する)。

### 段4: スペック準拠検証 (サブエージェント並列起動)

1. **単一メッセージで**以下のエージェントを並列起動する(直列に待たない):
  - `implementation-validator`(常時):
    - `subagent_type`: "implementation-validator"
    - `description`: "Spec compliance validation"
    - `prompt`: "ステアリングディレクトリ `[ステアリングディレクトリパス]` のスペック(requirements.md / design.md / tasklist.md)と実装の整合性を検証してください。対象ファイルは `[実装したファイルのパスリスト]` です。"
  - `doc-reviewer`(今回の変更が `docs/` 配下の永続ドキュメントを含む場合のみ):
    - `subagent_type`: "doc-reviewer"
    - `description`: "Document quality review"
    - `prompt`: "今回変更したドキュメント `[変更したdocs/のパスリスト]` の品質をレビューしてください。"
2. 両方の結果を待ち、指摘を統合して対応する。スペック乖離は実装を修正するか、意図的な乖離として `tasklist.md`(および必要なら `design.md`)に理由を記録する。

**注記**: ビルトインスキル(`verify` / `code-review`)が利用できない環境(このテンプレートの複製先等)では、当該段をスキップし、その旨を `tasklist.md` に記録して次の段へ進む。

**このステップが正常に完了したら、決して停止せず、ただちにステップ7に進むこと。**

## ステップ7: 振り返りとドキュメント更新

**このステップが消化するのは振り返りとドキュメント更新フェーズのチェックボックスである。** 各作業を終えた時点で該当行を `[x]` にする。

1. 今回の変更がプロジェクトの基本設計やアーキテクチャに影響を与えるか判断する。

2. 影響がある場合、`docs/`内の関連する永続ドキュメントを`Edit`ツールで更新する(不要と判断した場合はその判断を記録する)。

3. **このステップで `docs/` を更新した場合、その差分はステップ6の段3・段4のレビューを受けていない。** `doc-reviewer` サブエージェント(`docs/procedures/review-docs.md`)を更新差分に対して起動し、指摘対応まで終えてから当該チェックボックスを `[x]` にする(ステップ6へ戻ると振り返り→検証→振り返りの循環が生じるため、レビューをこのステップ内に閉じる)。

4. `Skill('steering')`を**振り返りモード**で実行し、`tasklist.md`に申し送り事項を記載する。
  - 実装完了日
  - 計画と実績の差分
  - 学んだこと
  - 次回への改善提案

**このステップの完了時点で、`tasklist.md` の `[ ]` は最終品質ゲート行の1件だけになる。**

**このステップが正常に完了したら、決して停止せず、ただちにステップ8に進むこと。**

## ステップ8: 最終品質ゲートとPR作成・worktree後片付け

最終品質ゲートは、実装・4段検証・振り返り・ドキュメント更新をすべて終えた後の**終端ステップ**である。最後の変更より後に実行されなければ最終ファイル状態の検証にならないため、必ずこの位置で実行する。

**ゲートは2回実行が前提**: ゲートに含まれるステアリング規律lint(C3: 未完了タスク検出)は、`tasklist.md` の最終品質ゲート行が `[ ]` である限り違反を報告するため、1回では緑にならない(自己参照)。C3にマーカー除外等の機構を足すと未完了タスクを隠す抜け穴になるため、決定論的な2回実行運用を正とする。

**ステップ4のG3受け入れ要否判定に従って分岐する**: 判定が「不要」なら**8-A**(既定)、「要」なら**8-B**を実行する。

### ステップ8-A: G3不要の場合(既定)

1. **ゲート1回目**: `Bash('uv run python3 scripts/local_quality_gate.py')` を実行する。C3が最終品質ゲート行の `[ ]` を検出して落ちるのは想定内である。ゲートは**最初の失敗で停止する**ため、steering lint(4/5)で止まり metered automation lint(5/5)には到達しない。1回目では**pytest / ruff / basedpyright がすべて緑であること**を確認する(全5検査の緑は2回目で確認する)。C3以外に違反があれば修正し、この手順を最初からやり直す。
2. `tasklist.md` の最終品質ゲート行を `[x]` にする(ゲート実行後に記録する)。
3. **ゲート2回目**: 同じコマンドを実行し、**全緑**を確認する。落ちた場合は修正し、手順1からやり直す。
4. 全変更をコミットする(Conventional Commits形式。本文に `Closes #[Issue番号]` を含める)。**完了マークやコミットは操作を実行した後に記録する**(記録と実態を乖離させない)。
5. ブランチをpushし、`gh pr create` でPRを作成する(本文: 概要 / 変更理由 / 変更内容 / テスト結果 / 関連Issue)。
6. worktreeを使用した場合は元のディレクトリに戻り、後片付けする:
  - `ExitWorktree` ツールが利用可能ならそれを使用する。
  - なければ `git worktree remove [worktreeパス]` を実行する(ブランチはPRマージまで残す)。
7. PRのURLをユーザーに報告する。マージはユーザーの判断に委ねる。

### ステップ8-B: G3受け入れが必要な場合

G3(`docs/procedures/harness-acceptance.md`)は「成功済みローカル品質ゲート」「固定commit」「clean clone」を実施条件とする一方、結果は記録として残さなければならない。この循環は、**受け入れ記録が製品ファイルを変更しない**ことを使って直線化する。以下の順序で実行する。

1. **候補ゲート**: 8-Aの手順1〜3をそのまま実行する(ゲート1回目 → 最終品質ゲート行を `[x]` → ゲート2回目で全緑)。
2. **候補コミット**: 全変更をコミットする(Conventional Commits形式。本文に `Closes #[Issue番号]` を含める)。これがG3の**固定commit**になる。
3. **G3実施**: 候補コミットを `/private/tmp` 等へclean cloneし、`docs/procedures/harness-acceptance.md` に従って**対話型のClaude Code IDEまたは対話型CLI**で受け入れを行う。**従量課金型headless modeを使わない**(`docs/external-automation-policy.md`)。
4. **結果記録**: 観察結果と証跡を、**複製ではなく元リポジトリ側**のステアリングディレクトリ(`.steering/[日付]-[タスク名]/`)の `acceptance-record.md` に記録する(複製内に作った確認用fixtureステアリングは複製ごと捨てる)。**この記録は `tasklist.md` のチェックボックスにしない**。
5. **最終ゲート**: 同じゲートコマンドを再実行し、**全緑**を確認する。最終品質ゲート行は手順1で既に `[x]` のため、自己参照が起きず**1回で緑になる**。
6. **記録コミット**: `acceptance-record.md` をコミットする(製品ファイルを変更しないため、手順3で観察した実機挙動は候補コミットのまま有効)。
7. push・PR作成(PR本文にG3の判定結果を含める)・worktree後片付け・URL報告は8-Aの手順5〜7と同じ。

**G3が不合格または保留だった場合**: 記録だけを追記して押し切らない。次の順序で戻る。

1. 4段検証フェーズのうち修正の影響を受ける段の行と、最終品質ゲート行を `[ ]` に戻す。
2. **ステップ6の該当段から再実行する**(再検証は増分方式に従い、修正の影響範囲に限定してよい)。
3. ステップ7で振り返りに不合格の経緯を追記し、**手順1(候補ゲート)からやり直す**。

**コミット・PR作成・G3受け入れ記録は `tasklist.md` のチェックボックス(C3対象)にしない**: PR前必須のゲートとPR作成が循環依存するため、実行管理はこの手順に委ねる。

## 完了条件

このワークフローは、以下の全ての条件を満たした時点で自動的に完了となる。
- ステップ0: worktree(またはフィーチャーブランチ)上で作業している。mainに直接コミットしていない。
- ステップ4.5: 計画がユーザーに承認されている(軽量パス適用時はその判定も、G3受け入れの要否判定も承認されている)。
- ステップ5: `tasklist.md`の**実装フェーズ**の全てのタスクが完了状態（`[x]`または正当な理由でスキップ）になっている。
- ステップ6: 4段検証(軽量パスでは縮約後の検証)をパスし、4段検証フェーズの全タスクが `[x]` になっている。
  - 段1: `uv run pytest` / `uv run ruff check` / `uv run basedpyright` がエラーなく成功
  - 段2: 変更種別に応じた実挙動(または成果物)の観察済み(純ドキュメントの場合はスキップを記録済み)
  - 段3: `Skill('code-review')` のfindingsに対応済み(軽量パスでは差分セルフレビュー済み)
  - 段4: `implementation-validator` のスペック準拠検証をパス(軽量パスではスキップと理由を記録済み)
- ステップ7: `tasklist.md`に申し送り事項が記載され、docs更新の要否を判断済みで、振り返りとドキュメント更新フェーズの全タスクが `[x]` になっている(この時点で残る `[ ]` は最終品質ゲート行の1件だけ)。
- ステップ8: 振り返り・docs更新を含むすべての変更より後に `uv run python3 scripts/local_quality_gate.py` が全体で1回パスし(8-Aでは2回実行の2回目が全緑、8-Bでは記録後の最終ゲートが全緑)、PRが作成され、URLが報告されている。G3が必要だった場合は `acceptance-record.md` に結果と証跡が記録されている。

この完了条件を満たすまで、自律的に思考し、問題解決を行い、作業を継続すること。
