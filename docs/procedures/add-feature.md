# add-feature 手順（機能追加ワークフロー）

新機能を既存パターンに従って実装するワークフローの手順書です。**計画はユーザー承認必須、承認後は完了まで自動実行**(「計画は承認必須・実装は自動」の原則)。ハーネス中立の正であり、各ハーネスの呼び出し口(Claude Codeでは `/add-feature` コマンド)はこの手順書に対応します。ハーネスが専用ツール(作業ツリー分離・調査/計画の委譲・構造化承認UI等)を提供する場合の割当はアダプタ側の定義を参照してください。

**引数**: 機能名(例: 「ユーザープロフィール編集」)

**承認ゲートはステップ4.5の1箇所のみ。** そこまでは調査・計画を進めて承認を求め、承認後はユーザーの介入なしに完了まで実行する。

## ステップ0: 作業の隔離

mainの作業ツリーを汚さないための担保。

1. 関連するGitHub Issueを確認する。存在しない場合は作成し、Issue番号を控える
2. 機能名から**英語 kebab-case のタスク名**を決める(例: 「ユーザープロフィール編集」→ `user-profile-edit`)。ステアリングディレクトリ名にも同じタスク名を使う
3. フィーチャーブランチ `feature/[タスク名]` を作成する。ハーネスが作業ツリー分離(worktree等)を提供する場合は併用を推奨する
4. ブランチ作成に失敗した場合(名前重複等)はユーザーに報告して中断する。自動リネーム等で暗黙に進めない

**mainブランチ上で実装を始めないこと**を必ず守る。

## ステップ1: 準備とコンテキスト設定

1. タスクコンテキストを確立する: 機能名 / 日付(YYYYMMDD) / ステアリングディレクトリパス `.steering/[日付]-[タスク名]/`
2. ステアリングディレクトリを作成する

## ステップ2: プロジェクト理解

1. AGENTS.md(と使用中ハーネスのアダプタ追記)を読み、プロジェクトの全体像を把握する
2. `docs/` の永続ドキュメントを確認し、関連する設計思想やアーキテクチャを理解する

## ステップ3: 既存パターンの調査

対象機能に関連する既存の実装パターン・命名規則・類似コンポーネントを検索・調査する。調査はトークン消費が大きく判断は浅いため、ハーネスが調査の委譲手段(軽量モデルへの委譲等)を提供する場合は委譲を推奨する。結論(発見したパターンの一覧と参照ファイルパス)を設計の入力とする。

## ステップ4: 計画フェーズ

1. 調査結果をもとに実装戦略(ステップごとの作業・変更対象ファイル・アーキテクチャ上のトレードオフ)を立案する。ハーネスが計画の委譲手段を提供する場合は利用してよい
2. **パス判定**: 以下の適用基準を**すべて**満たす場合、軽量パスを選択できる(1つでも満たさなければ通常パス):
   - 既存パターンの踏襲のみで、新しいアーキテクチャ要素・新規依存を導入しない(例: 新規ライブラリ追加・新規レイヤーや抽象化層の導入・DBスキーマ変更があれば非適用)
   - 変更対象が3ファイル以下(テスト除く)
   - `docs/` 永続ドキュメントの更新が不要
   - データ形式・API契約の破壊的変更がない
3. steering 手順(`docs/procedures/steering.md`)の**モード1**で、ステアリングファイルを作成する
   - **通常パス**: `requirements.md` / `design.md` / `tasklist.md` の3ファイル
   - **軽量パス**: `requirements.md` / `tasklist.md` の2ファイル。requirements.md に「**軽量パス**: 適用」と基準4項目の判定結果を記録する。あわせてステップ2-3の読み込み・調査も関連ドキュメント・関連コードに限定してよい

## ステップ4.5: 計画の承認(このワークフロー唯一の承認ゲート)

1. `design.md`(軽量パスでは `requirements.md`)の要点を計画として提示する: 実装アプローチ / 変更対象ファイル / tasklist.mdのタスク一覧 / 検証方針
2. **軽量パスを選択した場合は、適用判定(基準4項目の判定結果)を計画と一緒に提示する**(暗黙に軽量化しない)
3. 選択肢(承認して実装開始/修正を指示する/中止する)を明確に提示して承認を求め、回答を待つ。ハーネスが構造化された承認UI(計画モード・選択式質問等)を提供する場合はそれを優先する
4. **承認された場合**: 以降は無停止で実行する。ただちにステップ5へ
5. **修正を指示された場合**: 指示をステアリングファイルに反映し、再度この承認ゲートを実行する

> **承認待ちとStopフックの関係**: この承認ゲートで提示するtasklistは作りたて(全タスク未着手)である。ステアリング規律のStopフック(`check_tasklist_complete.py`)は、最新tasklistに完了タスク(`[x]`)が1つも無い**未着手状態では fail-open** し、承認待ちの停止に割り込まない。着手後(1つ以上完了)に未完了を残して停止した場合のみブロックする。未着手のまま放棄した場合の検出は、PR前のローカル品質ゲート(steering lint C3)が担う。

## ステップ5: 実装ループ(tasklist.mdの完全消化)

**tasklist.mdの全タスクが `[x]` になるまで繰り返すループ処理。完了したら停止せず、ただちにステップ6へ進む。**

1. tasklist.mdを読み込む
2. 未完了タスク(`[ ]`)がなければステップ6へ。あれば次へ
3. 先頭の未完了タスクを1つ特定し、steering 手順の**モード2**で実装する(development-guidelines.md の規約を遵守)
4. 完了したら即座にtasklist.mdの該当行を `[x]` に更新する
5. 1に戻る

**例外処理**(ループ中に自律的に対処し、ループを継続する):
- **ルールA(タスクが大きすぎる)**: サブタスクに分割してtasklist.mdを更新し、継続する
- **ルールB(技術的理由で不要)**: `- [x] ~~タスク名~~ (理由: ...)` の形式で記録し、継続する
- **エスカレーション(軽量パスの基準逸脱)**: 軽量パスで実装中に適用基準4項目のいずれかを満たさなくなった(4ファイル目の変更が必要になった / 新規アーキテクチャ要素・依存が必要と判明した / docs更新が必要と判明した / 破壊的変更が必要と判明した)場合は、通常パスへ昇格する: design.md を追加作成し、requirements.md の軽量パス欄を「非適用へ昇格(理由)」に更新し、ステップ6は4段フルで実施する。昇格条件はこの4項目の逸脱のみであり、**基準を破ったまま軽量パスで押し切ることを禁止する**
- **禁止**: 「後でやる」による意図的スキップ、理由なき放置、ユーザーへの判断委ね

## ステップ6: 4段検証

全タスク完了を確認してから、以下を**段1→段2→(段3・段4は並列可)の順**で実行する。機械的に落ちるものを先に潰し、コストの高いLLM検証を後段に置く。段3・段4は互いに独立しているため並列実行してよい。

**再検証は増分方式とする**: 各段の指摘を修正したら、再検証は変更の影響範囲に限定する(該当テスト・該当指摘の解消確認のみ。全段のやり直しはしない)。フル検証はステップ8の最終品質ゲートが決定論的に強制するため、中間のフル再実行は冗長である。

**最終品質ゲートはこのステップでは実行しない**: ステップ7で振り返りとドキュメント更新を行うため、ここでゲートを実行しても最終ファイル状態の検証にならない。ゲートはすべての変更を終えた後の終端ステップ(ステップ8)に置く。

**軽量パスの縮約**: 段1・段2は必須。段3は同一セッションでの変更差分セルフレビューに縮約してよい(独立レビュー原則の意図的な例外。適用基準を満たす小規模差分では、誤り相関のリスクよりプロセスコスト削減を優先する)。段4はスキップできる(スキップした旨と理由をtasklist.mdに記録する)。

### 段1: 静的検証

AGENTS.md技術スタック固有層に定義された検証コマンド(テスト・リント・型チェック)をすべて実行し、パスすることを確認する。

### 段2: 実挙動検証

| 変更種別 | 観察方法 |
|---------|---------|
| 実行可能コード | 実際に起動し入出力を観察する(ハーネスの検証支援機能があれば利用) |
| 手順・定義ファイル(Markdown) | 定義を実データに対して1回実行し、成果物を観察する |
| 純ドキュメント | 観察対象がないため段2をスキップし、その旨をtasklist.mdに記録する |

テスト通過だけで済ませず、必ず実挙動(または成果物)を観察すること。

### 段3: コードレビュー

変更差分のコードレビューを実施する(ハーネスのレビュー支援機能があれば利用)。正当な指摘を修正する(再検証は増分方式に従い、修正の影響範囲に限定する)。

### 段4: スペック準拠検証

`docs/procedures/validate-implementation.md` の手順で、ステアリングファイルと実装の整合性を検証する。`docs/` の永続ドキュメントを変更した場合は `docs/procedures/review-docs.md` のレビューも行う。**いずれも著者と独立した文脈(別セッション・別実行主体)での実行を推奨**(著者の思い込みによる誤り相関を減らすため)。ハーネスが並列の委譲実行を提供する場合は並列化してよい。

スペック乖離は実装を修正するか、意図的な乖離として tasklist.md(必要なら design.md)に理由を記録する。

**完了したら停止せず、ただちにステップ7へ進むこと。**

## ステップ7: 振り返りとドキュメント更新

1. steering 手順の**モード3**で振り返りをtasklist.mdに記録する
2. 今回の変更がプロジェクトの基本設計やアーキテクチャに影響する場合、`docs/` の関連する永続ドキュメントを更新する

**このステップの完了時点で、tasklist.md の `[ ]` は最終品質ゲート行の1件だけになる。**

**完了したら停止せず、ただちにステップ8へ進むこと。**

## ステップ8: 最終品質ゲートとPR作成・後片付け

最終品質ゲート(`scripts/local_quality_gate.py`。実体は技術スタック固有層に定義された単一ゲート)は、実装・4段検証・振り返り・ドキュメント更新をすべて終えた後の**終端ステップ**である。最後の変更より後に実行されなければ最終ファイル状態の検証にならないため、必ずこの位置で実行する。

**ゲートは2回実行が前提**: ゲートにステアリング規律lint(C3: 未完了タスク検出)が含まれる場合、tasklist.md の最終品質ゲート行はゲートをパスするまで `[ ]` のままであり、その未完了をC3が検出するためゲートは1回では緑にならない(自己参照)。C3にマーカー除外等の機構を足すと未完了タスクを隠す抜け穴になるため、決定論的な2回実行運用を正とする。

1. **ゲート1回目**: ゲートを実行する。C3が最終品質ゲート行の `[ ]` を検出して落ちるのは想定内である。ゲートは**最初の失敗で停止する**ため、C3より後段の検査は1回目には到達しない。1回目では**C3より前段の検査(テスト・lint・型検査等)がすべて緑であること**を確認する(全検査の緑は2回目で確認する)。C3以外に違反があれば修正し、この手順を最初からやり直す
2. tasklist.md の最終品質ゲート行を `[x]` にする(ゲート実行後に記録する)
3. **ゲート2回目**: ゲートを実行し、**全緑**を確認する。落ちた場合は修正し、手順1からやり直す
4. 全変更をコミットする(Conventional Commits形式。本文に `Closes #[Issue番号]` を含める)。**完了マークやコミットは操作を実行した後に記録する**(記録と実態を乖離させない)
5. ブランチをpushし、PRを作成する(本文: 概要 / 変更理由 / 変更内容 / テスト結果 / 関連Issue)
6. 作業ツリー分離を使用した場合は後片付けする(ブランチはPRマージまで残す)
7. PRのURLをユーザーに報告する。マージはユーザーの判断に委ねる

**コミット・PR作成はtasklist.mdのチェックボックス(C3対象)にしない**: 本ステップのコミット・PR作成は、PR前必須の最終品質ゲート成功後に実行する手順アクションである。これらをtasklistのチェックボックスにすると、PR前ゲート(C3)がその未完了を検出し、PR作成とゲートが循環依存する。実行管理はこの手順書に委ね、tasklistには載せない。

## 完了条件

- ステップ0: フィーチャーブランチ上で作業している。mainに直接コミットしていない
- ステップ4.5: 計画がユーザーに承認されている(軽量パス適用時はその判定も承認されている)
- ステップ5: tasklist.mdの全タスクが完了状態(`[x]` または正当な理由でスキップ)になっている
- ステップ6: 4段検証(軽量パスでは縮約後の検証)をパスしている
- ステップ7: tasklist.mdに振り返りが記載され、docs更新の要否を判断済みである
- ステップ8: 振り返り・docs更新を含むすべての変更より後にローカル品質ゲートを全体で1回パスし(2回実行の2回目が全緑)、PRが作成され、URLが報告されている

この完了条件を満たすまで、自律的に思考し、問題解決を行い、作業を継続すること。
