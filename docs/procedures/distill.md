# distill 手順（蒸留）

進化のサイクルの「④蒸留」を半自動化する手順書です。`.steering/` の振り返りから知見を取り出し、platform-harness への環流候補を抽出します。**環流の最終判断(PR作成)は人間が行う。** ハーネス中立の正であり、各ハーネスの呼び出し口(Claude Codeでは `distill` スキル)はこの手順書を参照します。

## 処理手順

### 1. 処理済み一覧の取得

1. `.steering/distill-*.md` に一致する過去の蒸留レポートを列挙する
2. 最新レポートの「処理済みディレクトリ」セクションを読み、処理済み一覧とする
3. 過去レポートがなければ処理済みなし(全ディレクトリが対象)

### 2. 振り返りの収集

1. `.steering/[0-9]*-*/tasklist.md` に一致する作業ディレクトリを列挙する(`YYYYMMDD-` 接頭辞のみが対象。`example/` 等の規約外ディレクトリは含めない)
2. 処理済みを除いた各 tasklist.md の「実装後の振り返り」セクションを読む
3. 振り返りが未記入(プレースホルダー `{...}` のまま、またはtasklist.md不在)のディレクトリは「振り返り未記入」として記録し、**処理済みに含めない**(記入後の再実行で拾うため)

### 3. 知見の抽出

「学んだこと」「次回への改善提案」「計画と実績の差分」から、独立した知見を1件ずつ抽出する。1つの箇条書きに複数の知見が含まれる場合は分割する。

### 4. 環流3基準による分類

各知見に以下を順に適用する:

| 基準 | 問い |
|------|------|
| 普遍性 | どのプロジェクトでも有効か(特定の技術スタック・クラウドに依存しないか) |
| 再現性 | 同じ失敗・同じ改善が他のプロジェクトでも起こりうるか |
| 抽象性 | プロジェクト固有の値(URL・モデルID・リージョン等)を含まないか |

- **3つすべてYes** → 環流候補(platform-harnessへ)
- **普遍性のみNo** → 技術スタック層候補(AGENTS.md技術スタック固有層へ)
- **それ以外** → プロジェクト内知見(記憶層へ)

判定に迷う場合は保守的に「プロジェクト内知見」に分類し、判断根拠を備考に残す。

### 5. レポート出力

`.steering/distill-[今日YYYYMMDD].md` に以下の形式で書き出す。同日のレポートが既存の場合は内容を統合して上書きする。

```markdown
# 蒸留レポート YYYY-MM-DD

## 環流候補(platform-harnessへ)

### 候補1: [タイトル]
- 出典: .steering/YYYYMMDD-xxx/tasklist.md
- 分類根拠: 普遍性Yes / 再現性Yes / 抽象性Yes([一言の理由])
- 変更対象ファイル: [platform-harness内の相対パス]
- 変更案: [そのまま使える具体的な変更内容の下書き]
- 参照チェック: [移植物が環流先に存在しない概念・スキル・ツールを参照していないか。参照している場合は一般表現への置換案を記す]

## 技術スタック層候補(AGENTS.md技術スタック固有層へ)

- [知見と反映案](出典)

## プロジェクト内知見(記憶層へ)

- [知見](出典) — 保存推奨: [Yes/No と理由]

## 振り返り未記入

- .steering/YYYYMMDD-xxx/([状態: 未記入 / tasklist不在])

## 処理済みディレクトリ

- .steering/YYYYMMDD-xxx/
(過去レポートからの引き継ぎ分も再掲し、常に全量を保持する)

## 環流PR計画

### [安定識別子: Issue URL / branch名]
- Issue URL: https://github.com/OWNER/REPOSITORY/issues/NUMBER
- branch: feature/example
- 統合する候補: 候補1、候補2
- 責務: [このPRだけが担う変更責務]
- 予定変更ファイル:
  - path/to/file-a
  - path/to/file-b
- 予定集合検査: [PASS / FAILと比較対象]
- 実集合検査: [PR作成直前にPASS / FAILと比較対象を記入]
- 独立性: [単独検証・レビュー・ロールバック・任意順マージの確認]
```

**変更対象ファイルのパス**は platform-harness のリポジトリ構造(このリポジトリの `.claude/`・`AGENTS.md` 汎用層・`docs/procedures/` に相当)を前提に書く。

### 6. ユーザーへの報告

1. 環流候補の件数と要点を報告し、platform-harness へのPR作成の判断を委ねる
2. プロジェクト内知見のうちセッション横断で有用なものは、記憶層(ハーネスが提供する記憶機能。なければ `.steering/` の振り返り)への保存を提案する
3. 振り返り未記入があれば、記入後の再実行を促す

### 7. 環流PR計画と変更ファイル非重複検査

環流PRを作成するときは、候補を責務単位へまとめ、PR作成前に次の2段検査を行う。検査はローカルGitと `scripts/check_pr_file_overlap.py` だけで完結し、GitHub Actionsの自動run、GitHub API、LLM headless modeを必須経路にしない。

#### 7.1 activeな環流作業の棚卸し

ここでactiveとは、未マージPR、作業中のfeature branch、または次回の環流バッチへ採用済みのIssueを指す。単にopenなだけで未採用のbacklog Issueは比較対象へ含めない。

1. 未マージの環流Issue、PR、feature branchを列挙する。手動GitHub UI、承認済みfetch後のremote-tracking ref、既存のローカルrefのどれを情報源にしたか記録する
2. 各作業をIssue URLとbranch名で識別し、実変更ファイルまたは予定変更ファイルを収集する
3. 新しい候補と既存作業を、蒸留レポートの「環流PR計画」へすべて記録する
4. 棚卸し日時、共通baseと各branchのcommit OID、remote-tracking refの最終更新確認方法を記録する。ネットワークを利用できずrefの鮮度を確認できない場合は、その限界を記録してfail-closedにする
5. activeな環流作業が新規候補1件だけの場合も、「他のactiveな環流branchなし」と記録する。比較対象の棚卸しを省略したPASSにはしない

計画時の安定識別子はIssue URLとbranch名である。まだ存在しないPR番号を予測して、計画・本文・相互参照へ記載してはならない。

#### 7.2 第1段: 予定変更ファイル集合の検査

各PR計画の「予定変更ファイル」を名前付き集合として入力する。同じ名前へ複数のパスを指定できる。

パスはリポジトリルート相対のPOSIX表記とし、文字大小を区別する。空白を除去した後、先頭`./`、絶対パス、親参照`..`、末尾`/`、連続`//`、バックスラッシュを含む非正規表記を拒否する。同一パスは1件へ重複除去し、辞書順で比較・表示する。renameは旧パスと新パスの両方を変更集合へ含め、表記差やrenameによる偽陰性を避ける。

```bash
python3 scripts/check_pr_file_overlap.py \
  --set issue-a=AGENTS.md \
  --set issue-a=docs/harness-guide.md \
  --set issue-b=docs/procedures/distill.md
```

activeなPR計画が2件以上ある場合、全PR組合せの積集合が空集合の場合だけ合格とする。1件だけの場合はCLIを実行せず、棚卸しの対象・日時・情報源・commit OIDを記録し、「比較対象なし（PASSではない）」として第1段を完了する。チェッカーが重複した組とパスを表示した場合は不合格であり、次のいずれかを環流PR計画へ記録して、空集合になるまで再検査する。

1. **統合**: 同じ責務の候補を1つのPRへまとめる
2. **再分割**: 重複ファイルの変更を責務単位で一方へ寄せ、別PRから外す
3. **共通前提の先行正典化**: 共通前提だけを独立してplatform-harnessへマージし、更新後のmainから残りのPRを再計画する

「先にPR AをマージすればPR Bが動く」というスタックPRやマージ順固定は、非重複検査の代替にならず、不合格を合格へ変更しない。

#### 7.3 第2段: Git実変更ファイル集合の検査

各feature branchの実装後、PR作成直前に共通baseとの差分を再検査する。計画後に追加された変更ファイルをここで捕捉する。

```bash
python3 scripts/check_pr_file_overlap.py \
  --base origin/main \
  --ref issue-a=feature/a \
  --ref issue-b=feature/b
```

activeなfeature branchが2件以上ある場合、実集合も全PR組合せの積集合が空集合の場合だけ合格する。1件だけの場合はCLIを実行せず、第1段と同じ棚卸し証跡に実変更ファイルを加え、「比較対象なし（PASSではない）」として第2段を完了する。存在しないbase/ref、Git差分取得失敗、空の実集合は検査不能としてfail-closedにし、非重複PASSへ変換しない。重複時は予定集合と同じ3つの解消方法を適用し、実集合を再検査する。

#### 7.4 PR番号と独立性

1. 各PRは単独で検証・レビュー・ロールバックでき、他の環流PRと任意順でマージ可能にする
2. PRをすべて作成した後、実在するPR番号で相互参照をまとめて追記する
3. PR作成後に変更ファイルが増えた場合は、第2段の実集合検査を再実行する
4. 予定集合と実集合の結果を各PRの説明または環流PR計画へ残す

### 8. platform-harness正典から派生プロジェクトへの同期

派生プロジェクトで得た環流候補は、派生側とplatform-harness側へ並行実装しない。次の順序を正規ルートとする。

```text
派生プロジェクトの振り返り
  → distillで環流候補へ分類
  → platform-harnessでプロジェクト固有値を除去して中立化・実装
  → platform-harnessのPRをマージしてrelease
  → リリース済み正典を派生プロジェクトへ同期
  → 派生側でローカル品質ゲートと必要な対話型受け入れ
```

platform-harnessで固有値やハーネス差を正典コアから適切な固有層・アダプタへ押し出す中立化は、他の派生プロジェクトでも利用できるテンプレート化を兼ねる。

派生側の同期記録は、同期作業に対応する `.steering/[YYYYMMDD]-[task]/tasklist.md` の「受け入れ記録」へ次の形式で残す。

```markdown
### 受け入れ記録
- 同期元: platform-harness [release tagまたはcommit]
- 同期対象: [ファイルとセクション]
- 派生固有差分: [残したプロダクト固有層・技術スタック固有層・アダプタ差分]
- ローカル品質ゲート: [コマンド、日時、結果]
- 対話型受け入れ: [IDE / 対話型CLI、結果。不要なら理由]
```

同期元のPRマージとreleaseが完了するまでは、派生側を同じ改善の正典として先行更新しない。

## 実行タイミングの目安

- 複数の作業(PR)がマージされたプロジェクトの節目
- platform-harness の更新前の棚卸し
- ユーザーの明示的な指示
