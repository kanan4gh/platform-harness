# steering 手順（作業計画・実装・振り返り）

ステアリングファイル(`.steering/`)に基づいた実装を支援し、tasklist.mdの進捗管理を行う手順書です。ハーネス中立の正であり、各ハーネスの呼び出し口(Claude Codeでは `steering` スキル)はこの手順書を参照します。

## 大原則

- **tasklist.mdが正式なドキュメント**。ハーネス内部のタスク管理機能は補助であり、進捗は必ずtasklist.mdに反映する
- **全タスクが `[x]` になるまで作業を継続する**
- タスクの完了・スキップはリアルタイムに編集で記録する(まとめて更新しない)

### 規律の担保

本手順の規律は、プロンプトへの記載ではなく決定論的チェックが担保する:

- **ローカル品質ゲート(`scripts/local_quality_gate.py`)**: steering lintを含む決定論的検証をPRマージ前にローカル実行する(全ハーネス共通の必須最終ゲート)
- **GitHub Actions**: ローカル品質ゲートの任意の手動ミラーであり、自動起動や必須status checkを前提にしない
- ハーネスがセッション内の強制機構(作業終了のブロック等)を提供する場合はそれが併用される。実装は各ハーネスのアダプタを参照

チェックにブロックされた場合は、タスクを完了させるか、下記ルールA/Bを適用する。

### ルールA: タスクが大きすぎる場合

タスクを小さなサブタスクに分割してtasklist.mdに追加し、1つずつ完了させる。

### ルールB: 技術的理由によるスキップ

スキップが許されるのは技術的理由(実装方針の変更・アーキテクチャ変更・依存関係の変更により不要になった)のみ。「時間の都合」「難しい」等は不可。

```markdown
- [x] ~~タスク名~~（実装方針変更により不要: 具体的な技術的理由）
```

スキップ理由は振り返りセクションにも記録する。

## モード1: ステアリングファイル作成(計画)

1. 現在の日付で `.steering/[YYYYMMDD]-[タスク名]/` を作成する
   - 同日に複数のステアリングを作る場合、後発のディレクトリ名が先発より辞書順で後になるよう命名する(「最新ディレクトリ=名前降順の先頭」の規約で強制機構が対象を特定するため)
2. 永続ドキュメント(`docs/` の product-requirements / functional-design / architecture / repository-structure / development-guidelines)を読み、方針を理解する
3. 以下のテンプレートを読み込み、プレースホルダーを具体化してファイルを作成する:
   - `docs/procedures/templates/requirements.md` → `requirements.md`(関連Issue URLと**使用ハーネス**を必ず記載。蒸留時にハーネス差を分析するため)
   - `docs/procedures/templates/design.md` → `design.md`
   - `docs/procedures/templates/tasklist.md` → `tasklist.md`

   **軽量パス適用時**(add-feature 手順の適用基準4項目をすべて満たし、requirements.md に「**軽量パス**: 適用」と判定結果を記録した場合)は、design.md を省略して requirements.md + tasklist.md の2ファイル構成とする(steering lint の C1 は宣言を検出して design.md 省略を許可する)。設計判断が発生しないことが適用条件のため、設計の記述は requirements.md の実装対象セクションで足りる
4. tasklist.mdはrequirements.md / design.mdに基づいて詳細化する:
   - 実装可能なタスクのみをリストアップする(「将来やるかも」「検討中」は含めない)
   - フェーズ分け・実装順序・サブタスクを明確にする
   - 新規計画のスコープは依頼・Issue・requirements.mdへ固定し、受け入れfixture作成を製品機能全体の再計画へ拡大しない

## モード2: 実装

1. tasklist.mdを読み、次の未完了タスク(`[ ]`)を特定する
2. 実装する(docs/development-guidelines.md の規約に従う)
3. 完了したら**即座に**該当行を `[ ]` → `[x]` に更新する(サブタスクも個別に)
4. 1に戻る。フェーズ完了時はtasklist.md全体を読み直し、見落としがないか確認してユーザーに報告する

計画と実装が乖離した場合:
- 該当タスクに注釈を付ける(例: `- [x] タスク名（実装方法を変更: 理由）`)
- 必要なら新タスクを追加し、設計変更が大きい場合はdesign.mdも更新する

作業を中断する場合(ユーザー指示等):
- tasklist.mdに引用ブロック(`>`)で中断記録を残す: 中断時点・完了済みの範囲・未コミット変更の有無・再開位置
- 再開時は中断記録を起点に作業を続け、記録に再開した旨を追記する

## モード3: 振り返り

全タスクが `[x]` になったことを確認してから、tasklist.mdの「実装後の振り返り」セクションを更新する:

- 実装完了日
- 計画と実績の差分(計画と異なった点・追加タスク・ルールBのスキップ理由)
- 学んだこと(技術面・プロセス面)
- 次回への改善提案
- リリース判断(テンプレートの表に従って評価・提案。リリースタグの実績を確認してからバージョンを提案する)

振り返りは蒸留(`docs/procedures/distill.md`)の入力になるため、他プロジェクトでも通用する知見(普遍性・再現性・抽象性)を意識して書く。

記録後、ユーザーに報告する:
「振り返りをtasklist.mdに記録しました。内容を確認してください。」
