# validate-implementation 手順（スペック準拠検証）

実装が**今回のスペック(ステアリングファイル)どおりに作られているか**を検証する手順書です。ハーネス中立の正であり、各ハーネスの呼び出し口(Claude Codeでは `implementation-validator` エージェント)はこの手順書を参照します。

**実行の推奨形態**: 著者と独立した文脈(別セッション・委譲実行等)で行うこと。可能なら著者と別モデルが望ましい(誤り相関の低減)。

## 責務と非責務

**責務**: ステアリングファイルと実装の整合性検証のみ。

**非責務**(4段検証の段1〜3が担うため、指摘しない):
- コーディング規約・命名・コメント等のコード品質(段3: コードレビュー)
- テストの通過・カバレッジ(段1: 静的検証)
- 実挙動の確認(段2: 実挙動検証)
- セキュリティ・パフォーマンスの一般的な問題(段3)

## 入力

- ステアリングディレクトリパス(`.steering/[日付]-[タスク名]/`)
- 実装ファイルのパスリスト

## 検証プロセス

1. ステアリングディレクトリの `requirements.md` / `design.md` / `tasklist.md` を読む
   - **軽量パス適用時は `design.md` が存在しない**(`requirements.md` に `- **軽量パス**: 適用` の宣言がある)。ファイル欠落を指摘せず、観点2を下記のとおり読み替える
2. 実装ファイルと関連する永続ドキュメント(`docs/`)を読む
3. 以下の3観点で検証する

### 観点1: 要求の反映(requirements.md)

- 「実装対象の機能」の各項目が実装に存在するか
- 「受け入れ条件」の各項目を実装が満たすか(条件ごとに判定)
- 「スコープ外」に踏み込んだ実装をしていないか

### 観点2: 設計との整合(design.md)

- コンポーネント設計・実装の要点どおりに実装されているか
- 乖離がある場合、それが `tasklist.md` または `design.md` に変更として記録されているか
  - 記録済みの乖離 → 問題なし(意図的な設計変更)
  - 未記録の乖離 → 指摘対象

**軽量パス(design.md 不在)の場合**: `requirements.md` の「実装対象の機能」を設計の代替として、実装がその範囲どおりかを検証する。加えて、軽量パスの適用基準(既存パターンの踏襲のみ・変更対象3ファイル以下(テスト除く)・`docs/` 更新不要・破壊的変更なし)を実装が実際に守っているかを確認し、逸脱していれば「通常パスへ昇格すべき乖離」として指摘する。`design.md` が無いこと自体は指摘しない。

### 観点3: タスク消化の妥当性(tasklist.md)

本検証(段4)は add-feature 手順ステップ6の途中で走る。**この時点でtasklist.mdに未完了(`[ ]`)が残っているのは正しい状態である**ことを前提に判定する。

- **実装フェーズ**の全タスクが `[x]` になっているか(判定対象はここまで)
- ルールB(技術的スキップ)の理由が技術的に妥当か(「時間の都合」等の混入がないか)
- タスクの完了マークと実装の実態が一致しているか(マークだけで未実装のものがないか)

**検証時点で未完了が正当な後続タスク**(以下)は、乖離として報告しない:

- 4段検証フェーズの段4(=本検証)の行。指摘対応の完了後に `[x]` になる
- 4段検証フェーズの段3(コードレビュー)の行。段3と段4は並列実行が許されているため、本検証の時点で未完了でよい
- 振り返りとドキュメント更新フェーズ(ステップ7で消化)
- 最終品質ゲートフェーズ(ステップ8で消化)

**逆方向の乖離は報告する**: 最終品質ゲート行が既に `[x]` になっている場合、ステップ8より前にゲートを実行した(または実行せずに記録した)ことになる。最終状態の検証が成立しないため、**乖離として指摘する**。同様に、振り返りとドキュメント更新フェーズがステップ5の実装ループ中に先行消化された痕跡があれば指摘する。

## レポート形式

```markdown
## スペック準拠検証結果

### 対象
- ステアリング: [ディレクトリパス]
- 実装ファイル: [リスト]

### 総合判定: [✅ 準拠 / ⚠️ 要記録 / ❌ 乖離あり]

| 観点 | 判定 | 備考 |
|-----|------|------|
| 要求の反映 | [✅/⚠️/❌] | [受け入れ条件 n/m 件充足] |
| 設計との整合 | [✅/⚠️/❌] | [乖離の記録状況] |
| タスク消化の妥当性 | [✅/⚠️/❌] | [スキップ理由の妥当性] |

### 相違点(ある場合のみ)

**相違点1**: [内容]
- スペック: [記載箇所と内容]
- 実装: [実際の実装]
- 分類: [未実装 / 未記録の設計変更 / スコープ外への逸脱 / 不当なスキップ]
- 推奨: [実装を修正する / tasklist.md・design.mdに記録する]

### 次のステップ
1. [最優先の対応]
```

## 検証の姿勢

- 判定は必ずスペックの記載箇所を根拠として示す(憶測で判定しない)
- 「実装の方が良い」と感じても、未記録の乖離は乖離として指摘する(記録を促すのが役割)
- 記録済みの意図的な変更は尊重し、指摘しない
