# review-docs 手順（ドキュメント品質レビュー）

プロジェクトドキュメント(PRD・機能設計書・アーキテクチャ設計書など)の品質を評価し、具体的な改善提案を作成する手順書です。ハーネス中立の正であり、各ハーネスの呼び出し口(Claude Codeでは `/review-docs` コマンド+`doc-reviewer` エージェント)はこの手順書を参照します。

**実行の推奨形態**: 著者と独立した文脈(別セッション・委譲実行等)で行うこと。著者自身のレビューは「自然」と思い込んだ誤りを見逃しやすく、独立した実行主体(可能なら別モデル)が誤り相関を減らす。

## 手順の骨格

1. 対象ドキュメントの存在を確認し、種類を特定する(PRD / 機能設計書 / アーキテクチャ設計書 / リポジトリ構造定義書 / 開発ガイドライン / 用語集)
2. 構造が適切なテンプレートに沿っているか確認する
3. 下記5観点で内容を評価する(対象が大きい場合[目安: 300行超または複数ドキュメント]は、観点を「完全性・明確性」「一貫性・実装可能性・測定可能性」の2グループに分けて並列にレビューし、結果を統合してよい)
4. 改善提案を含むレビューレポートを作成し、要点をユーザーに報告する

## レビュー5観点

### 1. 完全性
- 必要なセクションが全て含まれているか / 各セクションに十分な情報があるか / 曖昧な表現がないか / 前提条件が明記されているか
- 評価: ✅完全 / ⚠️改善推奨(一部不足) / ❌不十分(重要な情報が欠落)

### 2. 明確性
- 用語が一貫して使われているか / 定義が明確か / 図表が適切か / 具体例があるか
- 評価: ✅明確 / ⚠️改善推奨 / ❌不明瞭(解釈の余地が大きい)

### 3. 一貫性
- 他のドキュメントと矛盾がないか / 用語・フォーマットが統一されているか / 数値や日付が整合しているか
- 評価: ✅一貫 / ⚠️改善推奨(軽微な不整合) / ❌不一致(重大な矛盾)

### 4. 実装可能性
- 実装に必要な情報が揃っているか / 技術的に実現可能か / リソース見積もりが妥当か / 依存関係が明確か
- 評価: ✅実装可能 / ⚠️改善推奨 / ❌不十分

### 5. 測定可能性
- 成功基準が測定可能か / パフォーマンス要件に具体的な数値があるか / テスト方法が明確か / 受け入れ基準が定義されているか
- 評価: ✅測定可能 / ⚠️改善推奨 / ❌不明確

## ドキュメント種別ごとの追加観点

| 種別 | 追加チェック項目 |
|------|----------------|
| PRD | ターゲットユーザーの明確さ / 解決する問題の具体性 / KPI定義 / 優先度(P0/P1/P2) / スコープ外の明示 |
| 機能設計書 | システム構成図 / データモデル / ユースケースのシーケンス / エラーハンドリング / API設計(該当時) |
| アーキテクチャ設計書 | 技術選定の理由 / レイヤー構造の明確さ / 測定可能なパフォーマンス要件 / セキュリティ / スケーラビリティ |
| リポジトリ構造定義書 | 構造の視覚化 / 各ディレクトリの役割 / 命名規則 / 依存関係ルール / スケーリング戦略 |
| 開発ガイドライン | コーディング規約の具体例 / Git運用ルール / テスト戦略 / レビュープロセス / 環境セットアップ手順 |
| 用語集 | 分類の適切さ / 定義の明確さ / 具体例 / 関連用語のリンク / 索引 |

## レポート形式

```markdown
## レビュー結果: [ドキュメント名]

### 総合評価
| 観点 | 評価 | スコア(1-5) |
|-----|------|--------|
| 完全性 | [✅/⚠️/❌] | [1-5] |
| 明確性 | [✅/⚠️/❌] | [1-5] |
| 一貫性 | [✅/⚠️/❌] | [1-5] |
| 実装可能性 | [✅/⚠️/❌] | [1-5] |
| 測定可能性 | [✅/⚠️/❌] | [1-5] |
**総合スコア**: [平均]/5

### 良い点
(最低3つ)

### 改善が必要な点

#### [必須] 重大な問題
**問題1**: [問題の説明]
- **箇所**: [セクション名または行番号]
- **理由**: [なぜ問題か]
- **改善案**: [具体的な改善方法]
- **例**: 改善前/改善後を並べて示す

#### [推奨] 改善推奨
**問題2**: [問題の説明]
- **箇所** / **理由** / **改善案**(同上)

#### [提案] さらなる改善
**提案1**: [提案内容]
- **メリット**: [この改善のメリット]
- **実装方法**: [どう改善するか]

### 参考資料
- [関連ドキュメント / ベストプラクティス]

### 次のステップ
1. [最優先で対応すべきこと] ...
```

## レビューの姿勢

- **建設的**: 批判ではなく、改善のための提案を行う
- **具体的**: 「わかりにくい」ではなく、「どこが」「なぜ」「どう改善するか」を明示
- **バランス**: 悪い点だけでなく、良い点も必ず指摘する
- **実用的**: 実際に実行可能な改善案を提示する
- **根拠**: 改善提案には必ず理由を添える
