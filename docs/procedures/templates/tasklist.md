# タスクリスト

## 🚨 タスク完全完了の原則

**このファイルの全タスクが完了するまで作業を継続すること**

### フェーズとステップの対応

このファイルのチェックボックスは、add-feature 手順の**消化するステップごとに排他的に分割する**。どのステップがどのフェーズを消化するかを一意に決めることで、実装ループが後続ステップの作業を先取りする矛盾を防ぐ。

| フェーズ | 消化する add-feature ステップ |
|---|---|
| 実装フェーズ（フェーズ1〜N。個数は作業ごとに可変） | ステップ5（実装ループ） |
| 4段検証フェーズ | ステップ6 |
| 振り返りとドキュメント更新フェーズ | ステップ7 |
| 最終品質ゲートフェーズ（1行） | ステップ8 |

**ステップ5の実装ループが消化するのは実装フェーズだけ**であり、4段検証・振り返り・ドキュメント更新・最終品質ゲートの各フェーズを先行して消化しない。それぞれ対応するステップで完了させる。

**コミット・push・PR作成・G3受け入れ記録はチェックボックスにしない**（下部の注記を参照）。

### 必須ルール
- **最終的に全てのタスクを`[x]`にすること**
- 「時間の都合により別タスクとして実施予定」は禁止
- 「実装が複雑すぎるため後回し」は禁止
- 未完了タスク（`[ ]`）を残したまま作業を終了しない

### 実装可能なタスクのみを計画
- 計画段階で「実装可能なタスク」のみをリストアップ
- 「将来やるかもしれないタスク」は含めない
- 「検討中のタスク」は含めない

### タスクスキップが許可される唯一のケース
以下の技術的理由に該当する場合のみスキップ可能:
- 実装方針の変更により、機能自体が不要になった
- アーキテクチャ変更により、別の実装方法に置き換わった
- 依存関係の変更により、タスクが実行不可能になった

スキップ時は必ず理由を明記:
```markdown
- [x] ~~タスク名~~（実装方針変更により不要: 具体的な技術的理由）
```

### タスクが大きすぎる場合
- タスクを小さなサブタスクに分割
- 分割したサブタスクをこのファイルに追加
- サブタスクを1つずつ完了させる

> 軽量パス適用時は段3を変更差分のセルフレビューに縮約してよく、段4はスキップできる（スキップの旨と理由を該当行に記録する）。

## フェーズ4: 振り返りとドキュメント更新（add-feature ステップ7で消化）

- [ ] `docs/` 永続ドキュメントの更新要否を判断し、必要なら更新
- [ ] README.md を更新（コマンド追加・オプション変更・動作変更がある場合は必須）
- [ ] 実装後の振り返りを記録（このファイルの下部）

## フェーズ5: 最終品質ゲート（add-feature ステップ8で消化）

- [ ] 最終品質ゲートを全体で1回パス（実行後に確認）: `{最終品質ゲートコマンド}`

> **最終品質ゲートはこのファイルで最後のチェックボックスであり、1行だけ置く。** ゲートは実装・4段検証・振り返り・ドキュメント更新をすべて終えた後の終端ステップ(add-feature 手順ステップ8)であり、最後の変更より後に実行されなければ最終状態を検証したことにならない。したがってフェーズ3・4のすべてのタスクより後に置き、これ以降にチェックボックスを追加しない。
>
> ゲートにステアリング規律lint(C3: 未完了タスク検出)が含まれる場合、この行自身が `[ ]` のためゲートは1回では緑にならない(自己参照)。運用は **2回実行が前提**: 1回目でゲート本体(テスト・lint・型検査等)の健全性を確認 → この行を `[x]` にする → 2回目で全緑を最終確認する。C3にマーカー除外等の機構を足すと未完了タスクを隠す抜け穴になるため、決定論的な2回実行運用を正とする。
>
> **コミット・PR作成・G3受け入れ記録はチェックボックスにしない。** Conventional Commits形式のコミット・PR作成・ハーネス実機受け入れ(G3)の結果記録は、いずれも最終品質ゲートの前後に置かれる手順アクション(add-feature 手順ステップ8)である。これらをC3対象のチェックボックスにすると、PR前ゲートとPR作成が循環依存するため、実行管理は手順書に委ねる。
