# ハーネス換装（Harness Swap）

_2026-07-14 ディスカッションのまとめ_

## 名前の定義

**ハーネス換装** = SDDハーネスの中立コアを抽出し、各ツール（Claude Code、Codex 等）向けアダプタを薄く保つ再編。

- 英語名: **Harness Swap**（名詞・取り組み名）
- 設計目標の形容: **hot-swappable**（プロジェクトを止めずに途中からハーネスを替えられる）
- 動詞としても使う: 「codexハーネスに換装して」= "swap to the Codex harness"

命名の経緯: 候補として「脱皮」「メタハーネス化」「正典化」「SDD可搬化」を検討した。「換装」は元々馬具（ハーネス）のメタファーと整合し、動詞として指示文にそのまま使える（UXになる）点で採用。

## 動機となるユーザー要望

1. **途中換装**: Claude Code用SDDハーネスで動いているプロジェクトを、途中からCodex用に切り替えたい
2. **併用**: プロジェクトに両方のSDDハーネスを備え、適宜どちらかを選択して進めたい

## 基本認識: SDDハーネスの大部分はハーネス非依存

ハーネスを「切り替える」問題ではなく、**もともと中立な資産と、ハーネス固有の薄い層を分離する**問題として捉える。

| 分類 | 資産 | 割合 |
|------|------|------|
| **中立（大部分）** | `docs/` 永続ドキュメント、`.steering/` 作業記録と運用ルール、SDDプロセス知識（承認フロー・環流・蒸留） | 大 |
| **ハーネス固有（薄い層）** | 指示ファイル形式（CLAUDE.md vs AGENTS.md）、スキル/カスタムプロンプト実装、ツール前提の記述（AskUserQuestion・Skillツール・サブエージェント）、MCP設定（.mcp.json vs config.toml） | 小 |

## 提案構成: 中立コア + 薄いアダプタ

```
┌─────────────────────────────────────────────┐
│  中立コア（SSOT）                            │
│  AGENTS.md + docs/procedures/*.md           │
│  （SDDプロセスの正典・手順書）               │
├──────────────────────┬──────────────────────┤
│  Claude Code アダプタ │  Codex アダプタ      │
│  CLAUDE.md            │  config.toml         │
│  （@AGENTS.md import  │  カスタムプロンプト   │
│   + Claude固有追記）  │                      │
│  .claude/skills/      │                      │
│  （手順書への薄いラッパ）│                    │
└──────────────────────┴──────────────────────┘
```

### 構成要素

1. **正典を `AGENTS.md` に寄せる** — 複数エージェントが採用する事実上の業界標準。中立コア（現CLAUDE.mdの汎用層+プロダクト固有層の大半）をAGENTS.mdに置き、CLAUDE.mdは `@AGENTS.md` インポート+Claude固有追記だけの薄いファイルにする
2. **スキルの手順本文を中立ファイルに出す** — `steering` 等の手順本文を `docs/procedures/*.md` に出し、Claudeスキルはそれを読む薄いラッパに、Codex側は同じファイルをカスタムプロンプトから参照。同じ手順書を両ハーネスが読むので乖離しない
3. **併用は規約だけ決める** — Claude Codeは `.codex/` を無視し、Codexは `.claude/` を無視するので技術的には共存できる。危険なのはディレクトリ衝突ではなく中身の乖離であり、SSOT+参照構造で構造的に防ぐ

### 併用時の運用規約（要検討）

- 同一ステアリングをどちらのハーネスでも継続できるよう、tasklist.md の更新形式を手順書側で固定する
- どちらのハーネスで作業したかを `.steering/` の記録に残す（蒸留時に「ハーネス差による失敗」を分析できる）

## 品質分析: 中立化で品質はどこまで保てるか

スキルが品質に寄与する経路を3層に分解すると、中立化への耐性がまったく違う。

| 層 | 内容 | 劣化度 | 対策 |
|----|------|--------|------|
| **① 手順知識** | 何をどの順でやるか（3ファイル構成、環流3基準、承認フロー文言） | ほぼ無劣化 | Markdownはどのハーネスでも同じに機能。手順書自体が環流の対象になり品質向上の余地すらある |
| **② トリガー** | 正しいタイミングで手順が読まれるか | 中程度の劣化 | Claudeスキルのロードはほぼ決定論的だが、中立化するとモデルの遵守頼みになる。**ユーザー起動のエントリポイント**（Codexカスタムプロンプトで `/steering` 相当）を両側に用意し、重要な手順ほど明示起動に寄せる |
| **③ 強制力** | 手順が守られるか | 最も劣化するが回収手段あり | 下表参照 |

### 強制力の中立的な代替

| Claude固有の装置 | 中立的な代替 | 劣化度 |
|------------------|--------------|--------|
| AskUserQuestionの承認ゲート | テキストで伝えて待機 | 小（ハードストップ→ソフト） |
| サブエージェント（doc-reviewer等） | 別セッションで手順を実行 | 中（手動になるが独立性は保てる） |
| フック（ツール実行への介入） | スクリプト+CIチェック | 場合による |

**鍵は決定論的チェックへの置き換え**。「requirements.mdにIssue URLがあるか」「tasklist.mdが全部チェック済みか」「ステアリングに3ファイル揃っているか」といった検証はlintスクリプトに落とせる。決定論的チェックは最も移植性の高い強制力で、どのハーネスでもCIで同じように効く。

### このプロジェクトが既に持つ中立化耐性

uroboros-neo は**プロセスの状態がモデルの文脈ではなくファイルに置かれている**（`.steering/` 3ファイル、tasklist.md進捗管理、docs/永続ドキュメント）。品質の土台がファイル側にあるほど、ハーネス（手）を替えても品質が落ちない。

### 正直に「保てないもの」

- サブエージェントの並列性・自動性（別セッション起動は人間の手間になる）
- 無停止実行の制御（add-featureの「承認後は完全に無停止」— ハーネスごとに承認モデルが違う）
- フックによるツールコール単位の介入（各ハーネスのランタイム機能なので移植不能。ただし本プロジェクトはフック依存が薄い）

### 見立てのまとめ

品質の大部分は「手順知識+ファイル成果物」で決まっており、そこは無劣化で持ち出せる。劣化するのは強制力の層で、機械的に検証できるものはスクリプト化で回収でき、残るのは「モデルが承認を待たずに進む」類のソフトな逸脱リスク。

さらに、**中立化の過程で強制力をスクリプトに落とす作業はClaude Code側の品質も上げる**。中立化は妥協ではなく、「LLM頼みの強制」を「決定論的な強制」に格上げする機会である。

## 途中換装の移行パス

1. **棚卸し**: CLAUDE.mdとスキル群からハーネス固有記述を洗い出す（想像より少ないはず）
2. **中立コア抽出**: AGENTS.md + 手順書ファイル群に再編、CLAUDE.mdを薄いアダプタ化
3. **Codexアダプタ作成**: config.toml、必要ならカスタムプロンプト
4. **小さいタスクで検証**: Codexで1つステアリングを回し、プロセスが崩れる箇所を発見・手順書に環流
5. **platform-harnessへの還元**: この再編自体が汎用層の改善。テンプレート側を「harness-agnostic core + tool別adapter」構成に更新する

## platform-harness への含意

ハーネス換装により、platform-harness は「Claude Code用テンプレート」から**「SDDハーネスの正典 + 各ツール向けアダプタ」**に格上げされる。[harness-engineering.md](harness-engineering.md) のプラットフォームハーネス/プロジェクトハーネスの2層構造に、「ハーネス中立コア/ツールアダプタ」という直交する軸が加わる。

## 関連動向: Omnigent（Databricks製 Meta-Harness）

_2026-07-14 追記。ユーザー提供の調査メモに基づく整理であり、一次情報は未確認。_

Omnigent は Databricks が OSS（Apache 2.0）として公開した **Meta-Harness**。Claude Code、Codex、Cursor、Pi など複数のAIエージェントを統合・制御・共有する共通実行基盤で、Databricks版はマネージドサービスとして Unity AI Gateway・Sandbox 等と統合される。

- Harness を「Agent Loop を実行する Runtime」と定義し、YAMLの `executor.harness: claude-sdk` を `codex` に差し替えるだけで同じAgent定義を別ランタイムで実行できる
- コンセプトは Combine（Agentの組み合わせ: Claudeで設計→Codexで実装→Piでレビュー）、Control（Spend Budget・Git Push制限・承認フローを**PromptではなくMeta-Harness層で**管理）、Collaboration（SessionのURL共有）

### 符合

- 「メタハーネス化」という命名候補（不採用）は、業界の **Meta-Harness** という語と一致していた。ただし下記の通りOmnigentのメタハーネスと本取り組みは別レイヤーの仕事であり、「換装」の採用は結果的に正しかった
- `harness:` の一行差し替えは、本取り組みが想定した「codexハーネスに換装して」というUXの、YAMLとしての製品化と同型

### 決定的な区別: Omnigentは「換装の機構（ドック）」、本取り組みは「換装可能な積荷の規格化」

Omnigentが中立化するのは **Runtime層**（Session・Tool接続・Policy・Governance）。本取り組みが扱うのは **リポジトリ層のSDDプロセス資産**（AGENTS.md・手順書・.steering/・承認フロー）。ドックがどれだけ立派でも、積荷がClaude専用の形（.claude/skills に溶接された手順）をしていれば載せ替えられない。逆に積荷が規格化されていれば、換装機構は手作りアダプタでもOmnigentでもよい。

品質3層分析へのマッピング:

| 層 | Omnigentとの関係 |
|----|------------------|
| ① 手順知識 | Omnigentは提供しない。SDDプロセスの中身の中立化は Omnigent があっても必要 |
| ② トリガー | Omnigentのagent定義がエントリポイントになり得るので部分的に補完される |
| ③ 強制力 | Omnigentの主戦場。「Promptではなく Meta-Harness層で管理する」は、本ドキュメントの「LLM頼みの強制を決定論的な強制に格上げする」のプラットフォームレベルでの実装 |

### 移行パスへの影響

1. **中立コア抽出（移行パス1〜2）の価値は不変、むしろ上がる** — 規格化された積荷は、Omnigent的なレイヤーが普及するほど価値が出る
2. **Codexアダプタの手作り（移行パス3）は将来置き換わる可能性** — 「Omnigent OSS版を換装機構として評価する」という選択肢が生まれた
3. **pre-PR lint の一部はPolicy層に吸収され得る** — ただしリポジトリ内lintは移植性が最も高いので、まずlint、成熟したらPolicy層へ、の順が安全

## 作業ロードマップ（2026-07-14 承認済み）

各弾が「GitHub Issue + ステアリング」1セットに対応する。実施順は **第1弾 → 第4弾 → 第2弾 → 第3弾 → 第5弾 → 第6弾**（lint前倒し案）。lintを前倒しするのは、第2弾以降の再編作業の回帰確認の安全網になるため。ただしlintのチェック項目は第1弾の棚卸し結果を反映する。

| 弾 | 内容 | 規模 |
|----|------|------|
| **第1弾** | 棚卸しと換装境界の設計。CLAUDE.md・全スキル・.mcp.json からハーネス固有記述を洗い出し、「中立コア/Claudeアダプタ/Codexアダプタ」の対応表を設計ドキュメント化 | 小 |
| **第4弾** | 決定論的チェック（pre-PR lint）。ステアリング3ファイル存在・Issue URL・tasklist完了状態等のlintスクリプト+CI。換装と独立に価値がある | 小〜中 |
| **第2弾** | 中立コア抽出 — AGENTS.md化。CLAUDE.mdを `@AGENTS.md` インポート+Claude固有追記の薄いアダプタに再編。Codexは登場させず、Claude Codeでの無劣化回帰確認まで | 中 |
| **第3弾** | スキル手順の中立化。手順本文を `docs/procedures/*.md` に抽出しスキルを薄いラッパ化。優先順位は steering → distill → add-feature | 中 |
| **第5弾** | Codexアダプタ作成と実地検証。config.toml・カスタムプロンプト・併用規約に加え、**Codexで小さいステアリングを1本実際に回す**検証まで含む。崩れた箇所は手順書に環流 | 中 |
| **第6弾** | platform-harnessへの還元。テンプレート側を「harness-agnostic core + tool別adapter」構成へ更新（別リポジトリ作業、第1〜5弾の蒸留後） | 中 |

## 今後の論点
