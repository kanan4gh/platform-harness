# 設計書

## アーキテクチャ概要

既存steeringの事実関係や文言には触れず、受け入れ条件6項目のMarkdownチェック状態だけを`[ ]`から`[x]`へ変更する。補正の根拠は同じ作業のtasklist、完了証拠、マージ済みPR #21に限定する。

```text
tasklist・完了証拠・PR #21
          ↓ 完了実態を照合
過去requirementsの6チェック項目
          ↓ [ ] → [x] のみ補正
記録上の完了状態を一致させる
```

## コンポーネント設計

### 1. 補正対象の過去requirements

**責務**:

- 作業時点の要求と受け入れ条件を記録する。
- 各受け入れ条件の充足状態をチェック欄で示す。

**実装の要点**:

- 対象は`.steering/20260715-zzzz-record-outfit-v1-3-0-rollout/requirements.md`の受け入れ条件6項目だけとする。
- 各行の`[ ]`を`[x]`へ変更し、項目の文言・順序・周辺記録は変更しない。

### 2. 今回のsteering

**責務**:

- Issue #22に基づく補正範囲、設計、実施結果を記録する。
- 補正が新たな実装判断ではなく、完了証拠との整合回復であることを明示する。

**実装の要点**:

- 今回の要求・設計・タスク・検証結果だけを記録する。
- 過去steeringのtasklistや完了証拠を再編集しない。

## データフロー

### 受け入れ条件の記録補正

```text
1. 過去tasklistの「受け入れ条件6/6件充足」と完了証拠を確認する。
2. PR #21がマージ済みであることを既存記録とGit履歴で確認する。
3. 過去requirementsの対象6項目だけを[x]へ変更する。
4. 差分検査で文言・対象外ファイルが不変であることを確認する。
5. Issue #22を参照するPRを作成し、PR URLを完了記録へ反映する。
6. 全タスクを完了状態にして、steering lintを含む単一ローカル品質ゲートを実行する。
```

## エラーハンドリング戦略

- tasklistまたはGit履歴から完了根拠を確認できない項目があれば、その項目を`[x]`へ変更せず作業を中断する。
- 対象6項目以外の差分が生じた場合は、意図しない差分を特定して補正範囲へ戻す。
- 品質ゲートが失敗した場合は、原因を修正してから再実行し、失敗状態のままPRを作成しない。

## テスト戦略

### 静的差分検査

- 対象requirementsの変更行が6つのチェック状態だけであることを`git diff`で確認する。
- 今回のsteering記録を除き、変更ファイルが対象requirementsだけであることを確認する。
- `rg`で対象6項目がすべて`[x]`になったことを確認する。

### 統合検証

- `uv run python3 scripts/steering_lint.py`
- `uv run python3 scripts/local_quality_gate.py`

## 実装の順序

1. 今回の`tasklist.md`を作成し、ユーザー承認を得る。
2. 過去requirementsの6項目を`[x]`へ補正する。
3. 差分の限定性と完了状態を静的に検証する。
4. 振り返りを記録し、Issue #22を参照するcommitとPRを作成する。
5. PR URLと完了証拠を反映し、全タスクを完了状態へ更新する。
6. steering lintを含む単一ローカル品質ゲートを実行し、最終記録をpushする。
