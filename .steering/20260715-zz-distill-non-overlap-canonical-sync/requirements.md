# 要求内容

## 概要

`distill`で複数の環流候補をPRへ分割するとき、変更ファイル集合の重複をローカルで機械検査し、PR番号やマージ順へ依存しない計画を作れるようにする。改善はplatform-harnessで中立化・リリースした後、派生プロジェクトへ同期する正規ルートを手順として定義する。

- **関連Issue**: https://github.com/kanan4gh/platform-harness/issues/15
- **使用ハーネス**: Codex

## 背景

現行`docs/procedures/distill.md`は、複数の環流PRで変更ファイルを重複させず、PR番号は作成後に追記するよう注記している。しかし、どの集合を比較するか、空集合をどう確認するか、重複が見つかった場合にどう再分割するかが定義されていない。このため、スタックPRや番号の事前予測による相互参照事故を手順だけでは防ぎきれない。

また、派生プロジェクトで得た改善を派生側とplatform-harness側へ並行して実装すると、どちらが正典か曖昧になり同期差分が再発する。project-uroboros-neoで実証した「platform-harnessへ先に中立化して適用し、リリース済み正典を派生プロジェクトへ還す」順序を標準化する必要がある。

## ユースケースの軸

ハーネス保守者が、複数の環流候補を互いに独立してマージ可能なPRへ分割し、platform-harnessで確定した正典を派生プロジェクトへ安全に同期できる。

## 代表シナリオ

### シナリオ1: 既存の環流PRがある状態で新しい改善PRを計画する

次の環流PRが未マージで存在するとする。

| 安定識別子 | 状態 | 実変更ファイル集合 |
|---|---|---|
| Issue A / `branch-a` | PR作成済み | `AGENTS.md`、`docs/harness-guide.md` |
| Issue B / `branch-b` | PR作成済み | `docs/procedures/steering.md` |

新しい候補Cは`docs/procedures/distill.md`と`AGENTS.md`を予定している。PR作成前の比較結果は次になる。

```text
A ∩ C = { AGENTS.md }  -> FAIL
B ∩ C = {}             -> PASS
```

候補CはそのままPRにしない。`AGENTS.md`変更がAと同じ責務ならAへ統合し、分離可能ならCから`AGENTS.md`を外す。共通前提が必要なら、その前提だけを先に正典へマージした後、新しいmainを基準に残りのPRを再計画する。「Aを先にマージすればCも動く」というマージ順固定は非重複検査の代替にしない。

### シナリオ2: PR化前の環流候補をまとめて処理する

複数の候補が蒸留レポートに溜まっている場合、予定変更ファイルと責務からPR単位を組み立てる。

| 候補 | 予定変更ファイル |
|---|---|
| 候補1 | `.kiro/README.md`、`docs/harness-guide.md` |
| 候補2 | `.kiro/agents/sdd.json` |
| 候補3 | `docs/harness-guide.md`、`AGENTS.md` |
| 候補4 | `scripts/local_quality_gate.py` |
| 候補5 | `tests/scripts/test_local_quality_gate.py` |
| 候補6 | `docs/procedures/distill.md` |

候補1〜3はKiro対応という同じ責務で、候補1と3が`docs/harness-guide.md`で重複するため1つのPRへ統合する。候補4と5は実装と対応テストなので品質ゲートPRへまとめる。候補6は独立したdistill改善PRにする。

```text
Kiro対応PR ∩ 品質ゲートPR = {}
Kiro対応PR ∩ distill改善PR = {}
品質ゲートPR ∩ distill改善PR = {}
```

全組合せが空集合になった3PRは、任意順でマージできる。Issue #14をKiro・品質ゲートの機能PR、Issue #15をdistill手順改善PRへ分離した今回の進め方が実例である。

### シナリオ3: 計画時にはなかった重複が実装中に増える

計画時にPR Aを`docs/procedures/distill.md`、PR Bを`AGENTS.md`だけとした場合、予定集合は重複しない。しかしPR Aの実装中に`AGENTS.md`も変更すると、実Git差分の比較は次になる。

```text
PR A ∩ PR B = { AGENTS.md }  -> FAIL
```

この事故を捕捉するため、ブランチ作成前の予定集合と、PR作成直前のbaseに対する実変更集合を別々に検査する。実集合で重複した場合も、PR作成前に統合または再分割して解消する。

### シナリオ4: 派生プロジェクトの改善を正典化して戻す

派生プロジェクトで見つかった改善を派生側とplatform-harness側へ並行実装せず、次の順序で扱う。

```text
1. 派生プロジェクトの振り返りで改善を発見
2. distillで環流候補に分類
3. platform-harnessで固有値を除去して中立化
4. platform-harnessのPRをマージしてリリース
5. リリース済み正典を派生プロジェクトへ同期
6. 派生側のローカル品質ゲートと必要な対話型受け入れを実行
```

同期記録には、同期元のrelease tagまたはcommit、同期対象ファイル、派生側に残すプロダクト固有層・技術スタック固有層・アダプタ差分を記載する。固有値を正典コアから押し出して中立化した成果物は、他の派生プロジェクトでも再利用できるテンプレートになる。

## 実装対象の機能

### 1. 環流PR変更ファイル集合の非重複検査

- 環流候補ごとに予定変更ファイル集合を記録し、PR計画時に候補間の積集合が空であることを確認する。
- ブランチ実装後はbaseとの差分から実変更ファイル集合を取得し、複数PR間で再度比較するローカル検査手順を定義する。
- 比較入力、正規化、空集合の合格条件、重複パスの表示を決定論的に定義する。
- 重複が見つかった場合は、候補の統合、責務単位での再分割、共通前提の先行正典化のいずれかで解消し、マージ順依存のスタックPRを合格扱いにしない。
- GitHub Actions、GitHub API、LLM headless modeを必須経路にせず、ローカルGitと静的手順で完結させる。

### 2. PR番号・マージ順非依存の環流計画

- PR番号を事前予測して本文や設計へ固定しない。
- Issue URLとブランチ名を計画時の安定識別子とし、PR相互参照は全PR作成後に実在番号で追記する。
- 各PRが単独で検証・レビュー・ロールバックでき、任意順でマージ可能であることを完了条件にする。

### 3. platform-harness正典から派生プロジェクトへの同期ルート

- 派生プロジェクトの振り返りから抽出した知見を、まずplatform-harnessでプロジェクト固有値から中立化する。
- platform-harnessのfeature branchで実装・検証し、PRマージとリリースを完了してから派生プロジェクトへ同期する。
- 派生側の同期では、参照したplatform-harnessのリリースタグまたはcommit、同期対象、派生側に残す固有差分を記録する。
- 中立化によって固有値やハーネス差を正典コアから押し出すことが、そのまま再利用可能なテンプレート化になる原則を明記する。
- 同期後は派生プロジェクト側のローカル品質ゲートと、変更内容に応じた対話型受け入れを実行する。

### 4. 手順の回帰防止

- `docs/procedures/distill.md`へ非重複検査と正典同期ルートを追加する。
- 現行のPR作成注記を新手順へ統合し、番号予測禁止と任意順マージの規則を重複・矛盾なく保つ。
- 構造テストで、予定集合と実集合の2段検査、空集合の合格条件、重複時の解消、PR番号禁止、正典同期順序、ローカル検証境界を固定する。

## 受け入れ条件

### 非重複検査

- [ ] 環流候補ごとの予定変更ファイル集合をレポートへ記録する形式が定義されている。
- [ ] PR計画時の予定集合と実装後の実集合を別々に比較する手順がある。
- [ ] 全PR組合せの積集合が空である場合だけ合格とする規則がある。
- [ ] 重複パスを具体的に表示し、統合・再分割・共通前提の先行正典化で解消する規則がある。
- [ ] スタックPRやマージ順の固定を非重複検査の代替にしない。

### PR識別と独立性

- [ ] PR番号を作成前に予測しないことが明記されている。
- [ ] 計画時はIssue URLとブランチ名を安定識別子として使う。
- [ ] PR相互参照は全PR作成後に実在番号で追記する。
- [ ] 各PRが単独検証・レビュー・ロールバックでき、任意順でマージ可能であることを確認する。

### 正典同期

- [ ] `派生プロジェクトの知見 → platform-harnessで中立化・実装 → PRマージ・リリース → 派生プロジェクトへ同期`の順序が明記されている。
- [ ] 同期元のrelease tagまたはcommit、同期対象、派生固有差分を記録する規則がある。
- [ ] 中立化がテンプレート化を兼ねる原則が説明されている。
- [ ] 同期後に派生側のローカル品質ゲートと必要な対話型受け入れを実行する。

### 検証

- [ ] 手順の必須要素を検証する構造テストが追加される。
- [ ] Issue #15の変更ファイル集合が他の未マージ環流PRと重複しないことを確認する。
- [ ] `uv run python3 scripts/local_quality_gate.py`が合格する。
- [ ] 検証中のGitHub Actions自動runとLLM headless mode起動が0件である。

## 成功指標

- 複数環流PRの全組合せについて、変更ファイル集合の積集合が空であることをローカルで再現可能に確認できる。
- PR番号の事前予測とマージ順依存を0件にする。
- 派生プロジェクトへの同期元をrelease tagまたはcommitで一意に追跡できる。
- 新規・既存を含むローカル品質ゲートがすべて合格する。

## スコープ外

以下はこのフェーズでは実装しない:

- GitHub ActionsによるPR重複検査の自動起動
- GitHub APIへ依存する必須検査やbranch protection設定
- PRの自動作成・自動マージ・PR番号の自動予約
- 汎用的なリポジトリ同期CLIやファイル自動上書きツール
- project-uroboros-neo等の派生プロジェクトへの実同期
- Issue #15と無関係なdistill分類基準や他ハーネスアダプタの変更
