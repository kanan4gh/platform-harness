# 要求内容

## 概要

add-feature 手順のステップ5〜8・tasklistテンプレート・validator・ハーネス実機受け入れ(G3)の間に残っている実行順序の循環と矛盾を解消し、順序整合を決定論的テストで固定する。

- **関連Issue**: https://github.com/kanan4gh/platform-harness/issues/36
- **使用ハーネス**: Claude Code
- **軽量パス**: 非適用

## 背景

v1.4.1(`f313d8e`)で終端品質ゲートの**配置**は修正されたが、その前後にある実装ループ・4段検証・振り返り・G3実機受け入れとの**順序関係**が未整合のまま残っている。派生プロジェクト(project-uroboros-neo Issue #35)へのdirect-syncレビューで再現元が確認された。

### 問題1: ステップ5 / 6の完了条件が終端ゲート契約と矛盾する

- `docs/procedures/add-feature.md` ステップ5は「tasklist.mdの**全タスク**が `[x]` になるまで」ループすると定義する
- 同ステップ6も「全タスク完了を確認してから」を開始条件にする
- 一方 `docs/procedures/templates/tasklist.md` と同手順ステップ7 / 8は、最終品質ゲート行を最後の1件として**ステップ8まで `[ ]` のまま残す**契約である
- したがってステップ5のループは終了条件を満たせない。仮に「最終ゲート行以外」を消化すると解釈しても、標準tasklistには4段検証・docs更新・振り返り自体がチェックボックスとして存在するため、正式なステップ6 / 7より**先に**それらを実行してしまう
- `.claude/commands/add-feature.md` にも同じ矛盾がある
- `docs/procedures/validate-implementation.md` 観点3も「全タスクが `[x]`」を要求するため、段4実行中の**正しい**未完了状態を必ず乖離として誤報する

### 問題2: G3実機受け入れと最終状態ゲートが循環する

- `docs/procedures/harness-acceptance.md` はG3実施前に「ローカル品質ゲート成功済み」「対象commit / tagの固定」「使い捨て複製(clean clone)」を要求する
- G3結果を `acceptance-record.md` やtasklistへ記録すると、その記録変更はG3前に実行したゲートの検証対象外になる
- add-feature ステップ8は最終品質ゲート直後にcommit / push / PRへ進むため、**G3の実施・記録・再ゲートを置く正規の時点が存在しない**

## ユースケースの軸

**エージェント(および人)が add-feature 手順を上から順に実行するだけで、どのステップでどのチェックボックスを消化するかが一意に決まり、G3が必要な変更でも循環せずにPRまで到達できる。**

## 実装対象の機能

### 1. フェーズとステップの1対1対応(tasklistテンプレート)

- tasklistテンプレートの各フェーズ見出しに、消化を担当する add-feature ステップ番号を明記する
- 実装フェーズ / 4段検証フェーズ / 振り返り・ドキュメント更新フェーズ / 最終品質ゲートフェーズの4区分に整理する
- 最終品質ゲート行がファイル内で最後のチェックボックスである契約は維持する

### 2. ステップ5ループのスコープ限定

- ステップ5の終了条件を「**実装フェーズ**の未完了タスクがなくなったら」に限定する
- 後続フェーズ(4段検証・振り返り・最終ゲート)のチェックボックスをステップ5で先行消化することを禁止する
- 中立手順(`docs/procedures/add-feature.md`)とClaudeアダプタ(`.claude/commands/add-feature.md`)の双方に反映する

### 3. 後続ステップによるチェックボックス消化の明文化

- ステップ6が4段検証フェーズの各段を、ステップ7が振り返り・docs更新フェーズを、ステップ8が最終品質ゲート行を消化する
- ステップ6の開始条件を「実装フェーズの全タスクが `[x]`」に改める
- `docs/procedures/steering.md` モード3の確認条件を同じ区分で言い換える

### 4. validator観点3の誤報解消

- `docs/procedures/validate-implementation.md` 観点3を「実装フェーズの全タスクが `[x]`」に改める
- 検証時点で未実施が正当な後続タスク(段4自身・振り返り・ドキュメント更新・最終品質ゲート)を乖離として報告しない旨を明記する

### 5. G3受け入れの非循環順序(ステップ8の分岐)

- G3の要否判定基準を定義し、判定結果を requirements.md に記録する(判定はステップ4、実施はステップ8)
- **G3不要(既定)**: 現行フローを変更しない
- **G3必要**: 候補ゲート → 候補コミット(固定commit) → 対話型G3受け入れ → 結果記録 → 最終ゲート再実行 → 記録コミット → push / PR の順序を定義する
- `docs/procedures/harness-acceptance.md` に add-feature ステップ8との接続を明記する

### 6. 順序整合の決定論的テスト

- 手順書・テンプレート・validator・Claudeアダプタ・受け入れ手順の順序契約をpytestで固定する

## 受け入れ条件

### フェーズとステップの1対1対応

- [ ] `docs/procedures/templates/tasklist.md` の各フェーズ見出しに担当ステップ番号が記載されている
- [ ] 最終品質ゲート行がテンプレート内で最後のチェックボックスである

### ステップ5ループのスコープ限定

- [ ] `docs/procedures/add-feature.md` ステップ5の終了条件が実装フェーズ限定になっている
- [ ] `.claude/commands/add-feature.md` ステップ5が同じ終了条件になっている
- [ ] 後続フェーズのチェックボックスをステップ5で消化しない旨が両方に明記されている

### 後続ステップによる消化

- [ ] ステップ6 / 7 / 8がそれぞれ担当フェーズのチェックボックスを消化すると明記されている
- [ ] ステップ6の開始条件が「実装フェーズの全タスクが `[x]`」になっている
- [ ] `docs/procedures/steering.md` モード3の確認条件が同じ区分で書かれている

### validator観点3

- [ ] 観点3が「実装フェーズの全タスク」を対象にしている
- [ ] 段4自身・振り返り・docs更新・最終品質ゲートを乖離扱いしない旨が明記されている

### G3受け入れの非循環順序

- [ ] G3の要否判定基準が `docs/procedures/add-feature.md` に定義されている
- [ ] G3必要時の順序(候補ゲート → 候補コミット → G3 → 記録 → 最終ゲート → 記録コミット → PR)が定義されている
- [ ] G3不合格時のやり直し起点が定義されている
- [ ] G3不要時のフローが現行から複雑化していない
- [ ] `docs/procedures/harness-acceptance.md` にステップ8との接続が記載されている

### 決定論的テスト

- [ ] 上記の順序契約を検証するpytestが追加され、`uv run pytest` が緑になる
- [ ] `uv run python3 scripts/local_quality_gate.py` が全緑になる

## 成功指標

- 派生プロジェクトが v1.4.2 相当をdirect-syncしたとき、ステップ5の終了条件・validator観点3・G3の実施位置について追加の解釈を必要としない
- 順序契約を将来崩した場合、pytestが決定論的に検出する

## スコープ外

以下はこのフェーズでは実装しません:

- steering lint(C1〜C4)の判定ロジック変更(C3のマーカー除外等は抜け穴になるため導入しない)
- Stopフックの判定ロジック変更
- 実際のG3実機受け入れの実施。**G3不要と判定**する: 本変更はアダプタ構成・権限・フックのいずれも変更していない(`.claude/commands/add-feature.md` と `.claude/README.md` の**本文の手順記述のみ**を書き換えており、frontmatter・ファイル名・配置・`settings.json` の権限・`hooks/` の定義と登録はすべて不変)。判定基準は実装中に「実機挙動を変えるか」という結果ベースから「アダプタ構成・権限・フックを変更したか」という操作ベースへ精緻化した(振り返り参照)
- GitHub Actionsワークフローの変更
