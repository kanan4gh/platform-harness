# 設計書

## アーキテクチャ概要

本変更はコードではなく**手順契約**の設計変更である。中核は「tasklist.mdのチェックボックスを、消化するステップで排他的に分割する」こと。

現行は「tasklistの全チェックボックス」という1つの集合をステップ5が丸ごと引き受ける前提だったため、後続ステップ(6/7/8)が消化すべき項目まで巻き込み、終端ゲート行の存在で終了条件が自己矛盾していた。これを**フェーズ→担当ステップの全単射**に分割する。

```
tasklist.md のフェーズ            担当ステップ            消化タイミング
─────────────────────────────────────────────────────────────────
実装フェーズ(1..N)          →  ステップ5(実装ループ)    ループ内で逐次
4段検証フェーズ              →  ステップ6(4段検証)       各段の完了時
振り返り・docs更新フェーズ   →  ステップ7                各作業の完了時
最終品質ゲートフェーズ(1行)  →  ステップ8                ゲート1回目の後
─────────────────────────────────────────────────────────────────
コミット / push / PR / G3記録 → チェックボックスにしない(手順書が実行管理)
```

G3実機受け入れは、この分割の外側にある**別の循環**である。「ゲートは最終状態を検証する」「G3はゲート成功済みの固定commitに対して行う」「G3結果は記録として残す」の3条件は同時に満たせないように見えるが、**記録が製品ファイルを変更しない**ことを利用して直線化する。

```
候補ゲート(全緑) → 候補コミット(固定commit) → clean clone でG3
   → acceptance-record.md へ記録 → 最終ゲート再実行(全緑) → 記録コミット → push/PR
```

## コンポーネント設計

### 1. `docs/procedures/templates/tasklist.md`(中立テンプレート)

**責務**:
- フェーズ見出しに担当 add-feature ステップを明記し、消化主体を一意にする
- 冒頭の原則セクションに「フェーズとステップの対応」を追加し、ステップ5の消化範囲が実装フェーズ限定であることを宣言する

**実装の要点**:
- フェーズ番号は固定(1..2が実装例、3が4段検証、4が振り返り・docs更新、5が最終品質ゲート)。実装フェーズは可変個数のため「実装フェーズ(1..N)」と注記する
- 現行フェーズ3「品質チェックと修正」(テスト・リント)は4段検証の段1に統合する。同じ検証を2箇所に置かない
- 現行フェーズ4「ドキュメント更新」はステップ7の担当として残す
- 最終品質ゲート行が最後のチェックボックスである既存契約と、コミット/PRをチェックボックスにしない既存注記は維持する

### 2. `docs/procedures/add-feature.md`(中立手順書・正)

**責務**: ステップ5〜8の消化契約とG3分岐の定義。

**実装の要点**:
- **ステップ4**: パス判定に続けて「G3受け入れの要否判定」を追加し、結果を requirements.md に記録させる
- **ステップ5**: 終了条件を「実装フェーズに未完了タスクがない」に限定。後続フェーズの先行消化を明示的に禁止する
- **ステップ6**: 開始条件を「実装フェーズの全タスクが `[x]`」に変更。各段の完了時にその段のチェックボックスを `[x]` にする(段4は指摘対応の完了後)
- **ステップ7**: 振り返り・docs更新フェーズのチェックボックスを消化する旨を追加。「完了時点で `[ ]` は最終ゲート行1件」という既存の到達点はそのまま整合する
- **ステップ8**: 既定フロー(8-A)を現行のまま残し、G3必要時のみ 8-B を適用する分岐構造にする

### 3. `docs/procedures/validate-implementation.md`(段4の判定基準)

**責務**: 観点3の判定対象を、検証時点で完了しているべきタスクに限定する。

**実装の要点**:
- 「全タスクが `[x]`」→「実装フェーズの全タスクが `[x]`」
- 検証時点で未完了が正当な項目(段4自身・振り返り・docs更新・最終品質ゲート)を列挙し、これらを乖離として報告しないと明記する
- 逆方向の誤りも防ぐ: **最終品質ゲート行が `[x]` になっている状態は乖離**である(段4はゲート前に走るため)。この検出をルールに含める

### 4. `docs/procedures/harness-acceptance.md`(G3の実施手順)

**責務**: 実施条件(ゲート成功・固定commit・clean clone)が add-feature のどの時点で満たされるかを接続する。

**実装の要点**:
- 「add-feature ステップ8-Bとの接続」節を新設し、7段の順序を再掲する
- 記録の追加が製品ファイルを変更しないため、記録後の最終ゲートは1回で最終状態を検証できることを明記する(最終品質ゲート行は候補ゲート時点で既に `[x]`)
- 不合格時は候補ゲートからやり直す(記録だけを追記して押し切らない)

### 5. `docs/procedures/steering.md`(モード3の確認条件)

**責務**: 振り返り開始条件を新しいフェーズ区分で言い換える。

**実装の要点**:
- 「最終品質ゲート行を除く全タスク」→「実装フェーズと4段検証フェーズの全タスクが `[x]`」
- 振り返り・docs更新フェーズのチェックボックスはモード3の作業そのもので `[x]` になると明記する

### 6. ハーネスアダプタ

**責務**: 中立手順と同じ順序を、各ハーネスのツール割当で表現する。

**実装の要点**:
- `.claude/commands/add-feature.md`: ステップ5の終了条件・ステップ6の開始条件・ステップ7の消化・ステップ8のG3分岐を中立手順に合わせる
- `.claude/README.md`: 動作フロー(6〜9)・steeringスキルの振り返りモード起動条件・「全タスクが `[x]` になるまで」の設計原則を新しい消化契約へ更新する(段3レビュー指摘C1。旧契約のまま残すとアダプタ内部で矛盾する)
- `.agents/skills/add-feature/SKILL.md`(Codex) / `.kiro/skills/add-feature/SKILL.md`(Kiro): 薄いラッパのため、ステップ8-BのG3割当の追記と、8-A/8-B分岐に伴う既存行の改称に留める(構造は作り替えない)
- `docs/procedures/derived-project-rollout.md`: フェーズ4-5に同じG3循環(記録後の再ゲートがない)が残っているため、ステップ8-Bを参照させる(段3レビュー指摘A5)

### 7. `tests/procedures/test_add_feature_ordering.py`(新規)

**責務**: 順序契約の決定論的固定。

**実装の要点**:
- 既存テスト(`tests/adapters/test_harness_acceptance.py`)と同じ「該当セクションを切り出して文字列を検証する」スタイルに合わせる
- テンプレートについては構造的検証(最後のチェックボックスが最終品質ゲート行であること)も行い、文字列一致だけに依存しない
- 中立手順とClaudeアダプタが同じ順序契約を持つことを、両ファイルへ同一アサーションを適用して固定する

## データフロー

### 通常の変更(G3不要・既定)

```
1. ステップ5: 実装フェーズのタスクを逐次 [x]
2. ステップ6: 段1..段4を順に実行し、各段のチェックボックスを [x]
3. ステップ7: 振り返り・docs更新を実行し、当該フェーズを [x]
   → この時点で残る [ ] は最終品質ゲート行の1件のみ
4. ステップ8-A: ゲート1回目 → 最終ゲート行を [x] → ゲート2回目(全緑)
   → commit → push → PR
```

### ハーネス実機挙動を変える変更(G3必要)

```
1..3. 同上
4. ステップ8-B:
   a. 候補ゲート(1回目 → 最終ゲート行 [x] → 2回目全緑)
   b. 候補コミット(Conventional Commits / Closes #N) = G3の固定commit
   c. 候補コミットを /private/tmp へ clean clone し、harness-acceptance.md でG3実施
   d. 結果を .steering/[...]/acceptance-record.md へ記録(チェックボックスにしない)
   e. 最終ゲートを再実行(1回で全緑。ゲート行は既に [x] のため自己参照なし)
   f. 記録コミット(製品ファイルを変更しない)
   g. push → PR
   × 不合格: 実装を修正し a からやり直す
```

## エラーハンドリング戦略

手順書の変更のため例外機構はない。代わりに「誤った状態」を決定論的に検出する経路を定義する:

| 誤り | 検出経路 |
|---|---|
| ステップ5で後続フェーズを先行消化 | 段4(validator)が「最終品質ゲート行が `[x]`」を乖離として報告 |
| 後続フェーズの未消化のままPRへ | 最終品質ゲートのsteering lint C3 |
| 手順書・テンプレート・アダプタの順序が再び乖離 | `tests/procedures/test_add_feature_ordering.py` |
| G3記録後にゲートを再実行しない | harness-acceptance.md の接続節 + テスト |

## 実装の順序

1. 中立テンプレート(`templates/tasklist.md`)のフェーズ構造を確定する(他の変更が参照する語彙の起点)
2. 中立手順(`add-feature.md`)のステップ4 / 5 / 6 / 7 / 8を更新する
3. `validate-implementation.md` 観点3を更新する
4. `harness-acceptance.md` に接続節を追加する
5. `steering.md` モード3を整合させる
6. Claudeアダプタ(`.claude/commands/add-feature.md`)を更新する

## セキュリティ考慮事項

- G3手順の変更が、禁止されている従量課金型headless modeの起動を促さないこと。8-Bは対話型IDE / CLIでの実施のみを指す
- 候補コミットは push 前のローカルコミットであり、外部公開はステップ8-B(g)まで発生しない

## パフォーマンス考慮事項

- G3不要時のステップ数を増やさない(既定フローは現行と同一)
- 最終品質ゲートの実行回数は、G3不要時2回・G3必要時3回に固定する

## 将来の拡張性

- G3以外の外部受け入れ(実機デバイス検証等)が必要になった場合、8-Bの「候補コミット → 外部検証 → 記録 → 再ゲート → 記録コミット」というパターンを一般形として再利用できる
